WEBVTT Agora que nós finalizamos a configuração das nossas bottom tabs, antes de darmos início ao desenvolvimento da tela de home, nós precisamos fazer algo que é de suma importância e que nós não vamos poder seguir sem isso, que é a revalidação do token que temos aqui no nosso app. Basicamente... A autenticação através de tokens JWT é uma das formas mais comuns que se tem hoje em dia no mercado da comunicação entre o client-side e entre o service-side, no caso entre a aplicação web ou mobile, que é o nosso caso, com o back-end. É necessário muitas vezes uma credencial de acesso por motivos de segurança e essa credencial de acesso nada mais é do que o token que nós já estamos salvando dentro do nosso, da nossa Store do Zustand, e para nós recuperarmos é muito simples, vamos aqui mesmo dentro da nossa root layout, vamos recuperar um cost, igual a useUserStore, recuperar aqui o token, se nós dermos um console.log, e autenticarmos o nosso usuário, este arroba gmail.com, a senha, nós podemos ver o nosso Token sendo exibido aqui no nosso console. Esses tokens geralmente possuem um tempo de expiração relativamente curto por motivos de segurança. Até porque imagine que um token desse vaze. Supondo que um atacante tenha acesso a um desses tokens, ele não poderá fazer muita coisa com um token desse daqui vazado. Agora nós possuímos também um Refresh Token, que diferente do Token comum, o Refresh Token tem um tempo de expiração um pouco maior, porque a responsabilidade dele é literalmente fazer um par com o Token. E a partir do momento que o nosso Token for finalizado, nós vamos bater em uma rota específica lá no nosso back-end, onde nós vamos enviar o Refresh. E essa rota específica irá retornar para a gente um novo par de chaves, onde nós iremos realizar a substituição em tempo de execução da nossa requisição HTTP. na Store do Zustand e também na instância do Axis. Um detalhe interessante é que se nós pegarmos aqui no console o nosso token e virmos aqui na web, neste site de JWT.io e substituirmos neste campo este token que tem pelo nosso token, podemos ver que ele possui alguns dados específicos que estão criptografados dentro dele, no caso o ID do usuário, o e-mail e o momento em que este token irá expirar de fato. No caso este nosso token já está expirado, então ele é inútil pra gente. Consequentemente, se nós realizássemos uma requisição HTTP com ele, o nosso backend iria retornar um status code 401, contendo uma mensagem de que o token está expirado. Então, através destes parâmetros, a partir da próxima aula, nós vamos pegar e implementar a lógica da revalidação do token no nosso app.
