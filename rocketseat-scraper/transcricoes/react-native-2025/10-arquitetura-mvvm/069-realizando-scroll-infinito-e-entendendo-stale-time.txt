Consequentemente, a próxima vez que realizarmos uma requisição HTTP com esta mesma chave e com os mesmos parâmetros, ou seja, na mesma página que nós estamos buscando, Ao invés dele fazer a requisição assíncrona, ele irá simplesmente buscar os dados do cache interno do React Query, o que desta forma melhora bastante a experiência de usuário, diminuindo desta forma aqueles loadings que nós temos, uma aplicação que possui muitos loadings não é nem um pouco interessante a UX, a experiência de usuário fica um pouco defasada, fica um pouco chato, até porque o usuário vai ter que ficar esperando toda hora uma requisição ser feita. Uma outra coisa que é legal eu falar aqui para vocês é a questão do stale time que nós temos aqui no React Query. Esse stale time nada mais é o tempo esse cache vai ficar válido, o que significa nós temos de colocar aqui em milissegundos. Se nós colocarmos aqui, por exemplo, mil vezes 60 vezes 5, o cache da nossa query ficará válido por 5 minutos. Como eu disse, este tempo é com base em milissegundos, o que significa que nós temos aqui os segundos vezes 5. Se nós multiplicarmos 60 por 5, teremos 300, que é o total de segundos que 5 minutos tem. e nós pegarmos esses 300 e multiplicarmos por mil, teremos o resultado de milissegundos, o que significa que nós conseguimos controlar por quanto tempo um cache específico é válido. No caso, como nós estamos lidando aqui com a requisição de produtos, normalmente, nós podemos colocar um stale time maior. Por exemplo, de uma hora inteira. Então, no caso, nós substituiríamos aqui de 5 minutos para 60 minutos. Dessa forma, nós garantirmos que, num período de uma hora, toda requisição HTTP que for feita utilizando esta query específica, com os mesmos parâmetros, irão ser válidos por uma hora. E a próxima requisição que for feita utilizando este hook, Após o tempo do Style Time ser expirado, ou seja, daqui uma hora, de fato irá bater na nossa API e se houver alguma atualização nos produtos, nós de fato vamos ter acesso a essas atualizações.
