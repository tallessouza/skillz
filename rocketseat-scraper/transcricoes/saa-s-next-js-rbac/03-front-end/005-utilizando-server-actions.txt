WEBVTT Já nesse projeto uma das coisas que a gente vai fazer é começar a utilizar a nova funcionalidade do React integrado a frameworks que suportam ter um servidor próximo ao front-end, como por exemplo é o próprio Next, o próprio Astro e algumas outras opções, que é o uso de Server Actions, que é algo que está crescendo agora, mas que tem muitos pontos positivos dependendo do tipo de aplicação que você for construir. As Server Actions são basicamente formas da gente executar ações que vêm através de interações do usuário com a aplicação, porém do lado do servidor da nossa aplicação. Lembrando que esse servidor que eu tô falando não é o nosso back-end, a nossa API, e sim o servidor de Node integrado ao próprio Next.js ou, claro, aos outros frameworks que você possa estar utilizando ou pensando em utilizar. E quais são as vantagens reais disso, né? Da mesma forma que utilizar o conceito de Server Components diminui a quantidade de JavaScript enviado para o cliente da nossa aplicação, ou seja, para o navegador, para o browser, o uso de Server Actions tem o mesmo propósito. Quando a gente vai executar uma Server Action, essa Server Action, que nada mais é do que uma função, ela é executada como uma requisição HTTP. Ou seja, quando eu crio uma Server Action aqui dentro do React ou do Next, nesse caso, quando eu fazer o submit desse formulário, enviando para uma Server Action, na verdade, esse submit não vai ser feito simplesmente chamando uma função que vai estar escrita aqui dentro desse próprio componente. Na verdade, essa função vai se tornar um endpoint HTTP, ou seja, uma rota HTTP. E todo o código que a gente colocar nessa função, nessa Server Action, é algo que vai ficar exclusivamente no lado do servidor. Isso quer dizer que, estando do lado do servidor, a gente pode sim executar códigos que dependam de funcionalidades mais sensíveis da nossa aplicação, como, por exemplo, até um banco de dados. Mas, lembrando que 90% das vezes não é isso que você quer fazer. É muito comprovado já que, assim que uma aplicação cresce, a gente tem que ter o back-end separado do front-end, até para facilitar a manutenção, testes e tudo mais. Mas, mesmo assim, mesmo sem a gente executar ações sensíveis, como uma conexão com o banco de dados dentro da Server Action, a gente pode sim utilizar a Server Action com o propósito de diminuir a quantidade de JavaScript enviada para o client-side. Porque todo o código da Server Action, como ele vai funcionar como uma rota HTTP, esse código não vai ser enviado para o bundle do front-end, o código JavaScript necessário para o front-end funcionar ali dentro do navegador. E assim, se dentro dessa action nós utilizarmos bibliotecas de validação ou qualquer outro tipo de biblioteca que a gente possa utilizar num formulário, nada disso vai ser enviado ali para o front-end. Isso vai deixar nossa aplicação, claro, mais leve. Mas, lembrando que Server Action não é bala de prata. Nada que a gente está mostrando aqui, desde Server Components, Server Actions, tudo isso que está vindo de novo para dentro do React, nada disso elimina tudo o que a gente aprendeu com anos de React, até pelo nome do React, a reatividade do React. Então, a gente tem que evitar essa ideia de que tudo no React precisa, obrigatoriamente, agora utilizando o Next ou o Astro ou qualquer outro framework, ser executado pelo lado do servidor. Não! O React é uma biblioteca reativa. Então, a gente vai sim precisar ter coisas rodando do lado do cliente, JavaScript do lado do cliente. Senão, a nossa aplicação vai acabar se tornando uma aplicação igual a que a gente construía lá com PHP, com Ruby, antigamente. E esse não é o propósito, porque, justamente, React, ele existiu, ele veio a existir para trazer essa reatividade, para deixar nossa aplicação mais rápida a olho do usuário final.
