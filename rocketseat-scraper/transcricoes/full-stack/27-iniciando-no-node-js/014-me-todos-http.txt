Vamos entender então o conceito, o que são os métodos HTTP, porque a gente vai utilizar bastante eles na prática, tá? Então relembrando, nós temos essa relação cliente-servidor, lembrando que a nossa API está dentro de um servidor, e nessa relação cliente-servidor utilizamos o protocolo HTTP, que é o padrão de comunicação. E nessa relação da nossa aplicação, da nossa API com o cliente e do cliente com a nossa API, a gente tem basicamente duas direções. O nosso cliente pode fazer requisições para a nossa API, ou seja, pedidos, certo? Vamos, por exemplo, cadastrar um novo produto e a nossa API pode devolver e deve devolver respostas. Aqui tem uma informação importante, tá? Sempre, preste bastante atenção nisso, sempre a API tem que devolver uma resposta para o cliente, seja uma resposta positiva ou negativa, seja de sucesso ou seja de erro, o cliente sempre precisa ter uma resposta, porque se a API não devolve uma resposta para o cliente, o cliente vai ficar lá esperando, esperando, esperando, esperando, até estourar o timeout, ou seja, o tempo de tolerância de espera ali, por exemplo, do navegador. Então sempre a API tem que dar uma resposta, seja de sucesso ou de erro, por exemplo, digamos que o nosso cliente, que pode ser uma aplicação desktop, um front-end web, um aplicativo mobile e assim por diante, digamos que o nosso cliente solicitou aqui, fez uma requisição para a nossa API, solicitando a lista de produtos cadastrados e filtrados por uma característica, por exemplo, de cor, sei lá, produtos que são da cor vermelha. A API pode devolver aqui, como resposta, pode devolver aqui essa listagem dos produtos com essa característica. Então temos aqui uma resposta com sucesso, por exemplo, ó, tá aqui a lista dos produtos, tá tudo certo. Agora digamos que o usuário vai tentar criar aqui um novo produto, a API pode devolver, por exemplo, uma resposta de que o produto já está cadastrado, de que o produto já existe, ou pode devolver, por exemplo, a resposta de produto cadastrado com sucesso, ou digamos que às vezes o cliente vai tentar acessar um recurso que não existe, uma rota que não existe dentro da API, a API pode devolver uma resposta de erro dizendo, olha, não foi encontrado o famoso not found, tá, de que não foi encontrado, enfim, então sempre tem que ter uma resposta, tá. Então só relembrando essa relação do nosso cliente com a nossa API. Existe também uma outra característica importante que na requisição, sempre que o nosso cliente vai fazer uma requisição, ele também precisa dizer qual é a ação que ele vai executar, qual é a ação que ele quer executar dentro da API. E pra determinar, pra dizer qual é essa ação, nós utilizamos os métodos HTTP. Os métodos HTTP, eles são responsáveis por indicar qual é a ação que vai ser executada dentro da API. E basicamente temos o GET, que significa que a gente está querendo ler, ou seja, buscar, consultar alguma coisa na nossa API. O POST, basicamente significa que nós queremos criar alguma coisa lá na nossa API. Então perceba que esses métodos HTTP, eles são semânticos, eles dão um significado pra ação, identifica o que a gente quer fazer dentro da nossa API. Aqui a gente tem o PUT, que é quando a gente quer atualizar, por exemplo, quero fazer a atualização dos dados de um produto, o DELETE pra deletar, excluir, e o PET é quando eu quero fazer uma atualização parcial, por exemplo, eu quero atualizar apenas a foto do produto, ou apenas o status de disponível ou produto vendido, por exemplo. Então é quando eu quero atualizar uma coisa mais específica, aí a gente utiliza esse método PET. Então, basicamente aqui, cada método indica uma ação que a gente quer executar. E por que o método HTTP é tão importante? Olha só esse exemplo aqui, vai ficar mais claro. Porque a gente pode ter a mesma rota, perceba que a gente pode ter uma mesma rota que faz ações diferentes. Eu posso aqui acessar, nesse mesmo endereço aqui, https://meucervidor.com.br-products pra ler, ou seja, consultar os produtos cadastrados, e eu posso utilizar o mesmo endereço pra criar um novo produto. Como a API vai diferenciar quando eu quero ler os produtos cadastrados, ou seja, consultar, e quando eu quero criar um novo produto? A API vai diferenciar pelo método HTTP. Então quando eu digo assim, olha, eu estou fazendo aqui um GET nessa rota de produtos, significa que eu estou lendo, eu quero consultar. Agora, se eu faço um POST na rota de produtos, então significa que eu quero criar um novo produto. Perceba que aqui é onde o método HTTP é importante, identificar qual ação a gente quer fazer, porque a gente pode ter rotas com o mesmo nome, como a gente tem aqui, barra products em ambas, então o mesmo endereço, mas a gente utiliza o método HTTP pra diferenciar qual ação. Aqui eu posso fazer de leitura com o método GET, ele entende que eu quero fazer uma leitura, e o POST ele vai entender que eu quero fazer aqui, cadastrar, registrar um novo produto. É claro que quando a gente vai desenvolver, é a gente que vai dizer isso, olha, quando for GET eu quero ler, eu quero consultar, quando for POST é porque eu quero criar. Então nós mesmos vamos definir isso dentro da nossa API, tá? Então agora você já conhece rotas, já conhece aqui então os métodos HTTP, e os métodos HTTP a gente utiliza aqui na requisição pra indicar, deixar claro qual é a ação que a gente quer executar dentro da nossa API. E na resposta a gente pode também adicionar mais uma informação. Lembra que eu disse aqui pra você que uma resposta pode ou não ter conteúdo? Mesmo se a resposta não tem conteúdo, a gente pode sinalizar se deu certo ou não a requisição. Qual que é uma resposta com conteúdo? Por exemplo, uma resposta que devolve o ID do produto cadastrado, que devolve a listagem de produtos baseado em um filtro, e assim por diante. Uma resposta sem conteúdo pode ser uma resposta de uma requisição de deletar um produto, foi lá, executou, deletou o produto, e aí não precisa retornar os detalhes do produto, apenas se deu certo ou não, se deu ali certo ou seja deletado com sucesso, por exemplo. Mas aí a gente pode utilizar uma outra camada de informação, outro tipo de informação que são os HTTP Response Status Codes, ou apenas Status Codes. Os códigos de status de resposta HTTP, eles indicam o status de uma solicitação. Por exemplo, se uma solicitação HTTP específica foi concluída com êxito. A gente tem 5 faixas de status, então as que começam com o 1 são respostas informativas, então 100 e daí pra frente, as de 200 são de sucesso, as de 300 são de redirecionamento, as de 400, erro do cliente, e as de 500, erro no servidor. Então tem essas faixas 100, 200, 300, 400 e 500, então cada faixa tem um significado. E a gente utiliza o Status Codes ali na resposta pra indicar exatamente isso, aí a gente indica se deu sucesso, se deu erro, enfim, então mais uma camada de informação que a gente adiciona ali na resposta do usuário. Então eu posso devolver uma resposta ali pro meu usuário, ou melhor dizendo, a minha API pode devolver uma resposta pro cliente, e nessa resposta não precisa ter conteúdo, por exemplo, uma listagem ou informações de um produto, por exemplo, mas a gente pode utilizar um Status Code pra dizer se deu certo ou não, se deu erro, por exemplo, ou se, por exemplo, foi um erro do cliente ou um erro do servidor, enfim, aqui eu trouxe alguns exemplos, tá, aqui dentro de uma resposta de um Status Code informativo a gente pode retornar, por exemplo, 102 de processando, às vezes não vai ficar pronta a requisição naquele momento ali, você pode retornar um 102 que significa processando. Aqui a gente tem na de sucesso, por exemplo, algumas bem conhecidas como 200, que é a requisição bem sucedida, ou seja, deu tudo certo, a gente tem o 201 que é utilizado pra dizer que foi criado, por exemplo aqui, geralmente a gente utiliza após um post, então foi lá, fez o cadastro, 201 significa que foi criado com sucesso. E aqui o de 200 é um sucesso mais genérico, né, tipo, deu sucesso. Aqui a gente tem o de redirecionamento, tá na faixa do 300, 301 significa que foi movido permanentemente, o 302 foi movido, às vezes moveu alguma coisa de lugar, ou enfim, mudou o endereço e assim por diante, né. O de 400 é erro do cliente, erro do cliente é quando é um erro que foi causado por uma requisição feita pelo cliente, por exemplo, o 400, uma bad request, às vezes o cliente faz uma requisição que tá fora do padrão, o de 401, às vezes o usuário tá tentando acessar uma rota que ele não tem autorização, então é o famoso não autorizado. O de 404 é o famoso não encontrado, vou tentar acessar uma rota que não existe na API, então eu posso ter aqui um 404 de não encontrado. E aqui, erro no servidor, é quando é um erro gerado do lado do servidor, não é um erro ocasionado diretamente pelo cliente, né, por quem fez a requisição, mas é um erro mais interno do servidor, por isso tem aqui o 500, famoso erro interno. Você não precisa se preocupar em decorar todos esses status codes, não entra nessa neura de decorar isso, tá, com o passar do tempo você vai vendo os mais utilizados e vai ficando super natural, tá bom?
