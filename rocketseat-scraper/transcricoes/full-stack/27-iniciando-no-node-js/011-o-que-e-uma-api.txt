Então vamos primeiro compreender qual é o conceito, tá? Uma API, essa sigla, ela vem da combinação dessas palavras aqui, o A de Application, o P de Program e o I de Interface, ou seja, é uma interface de programação de aplicação que oferece funcionalidades sem a necessidade de saber como foram implementadas. O que isso significa? Imagina só, você está desenvolvendo uma aplicação para um consultório médico de agendamento de pacientes, você pode desenvolver uma API e nessa API desenvolver todas as funcionalidades de uma aplicação de agendamento para um consultório e você pode ter um front-end que é a aplicação web feita com o React, por exemplo, uma aplicação front-end ali, web, que pode consumir, utilizar os recursos de uma API, só que sem saber como essas funcionalidades foram implementadas, ou seja, você consegue separar a parte lógica da aplicação, todas as regras de negócio, ou seja, o back-end e desenvolver separado essas funcionalidades numa API que vai ser utilizado por um front-end. Estou mencionando aqui um front-end web, mas poderia ser desktop, poderia ser mobile, por exemplo, então essa é a ideia de uma API. Então você cria uma aplicação que disponibiliza as funcionalidades para ser utilizado por uma aplicação front-end, por exemplo, e a aplicação front-end, ela não sabe, não precisa saber como foi exatamente implementada essas funcionalidades do lado da API, basicamente existem regras, ou seja, um padrão de comunicação. E é aí que entra, então, aqui a estrutura de uma API que eu vou explicar para você agora. Basicamente, a API é composta por um cliente, a própria API e o servidor, o servidor é onde a API vai estar hospedada, é onde a API está armazenada, é onde a API está o servidor. O cliente pode ser um celular, uma aplicação mobile, o cliente pode ser uma aplicação web, o cliente pode ser uma aplicação desktop e assim por diante. Agora vamos utilizar aqui uma analogia, para deixar o mais didático possível, uma analogia de um restaurante. Pensa no cliente como sendo um cliente do restaurante. Quando o cliente chega no restaurante, ele vai olhar o menu, o cardápio, e vai então identificar o que ele deseja fazer, por exemplo, eu quero tal prato. O cliente faz uma requisição, ou seja, uma solicitação para o garçom, o garçom é a nossa API, o garçom conhece o cardápio, ele sabe de todos os pratos, ele sabe todos os detalhes de como esse prato é feito. Então o garçom é quem recebe a solicitação do cliente, ou seja, a requisição, e o garçom vai processar dentro do servidor essa requisição, que é a nossa cozinha, então ele vai lá, vai processar, ele vai fazer o prato, e quando pronto, ele vai ir lá devolver para o cliente a resposta, ou seja, olha, seu prato está aqui, está pronto, bom apetite, enfim, então o garçom volta ali com uma resposta que é o prato pronto. Agora vamos olhar para uma aplicação real, voltando no exemplo de uma aplicação de agendamento de um paciente. Digamos que o cliente aqui é uma aplicação web, um front-end web, e aí quando eu clico lá no botão de cadastrar um novo paciente, eu preenchi todas as informações do paciente, clico no botão de cadastrar, o front-end faz uma requisição para a API, e nessa requisição a gente tem a solicitação para fazer um cadastro com os dados do cliente que a gente quer cadastrar. A nossa API vai processar isso dentro do servidor, por exemplo, salvar no banco de dados os dados do paciente, vai verificar se o paciente já é um paciente que existe, por exemplo, se é um paciente já cadastrado, a nossa API pode devolver como resposta, por exemplo, para o cliente, cliente já cadastrado, ou esse cliente já existe, ou algo do tipo. Agora, se o cliente realmente não existe, o servidor pode devolver como resposta uma mensagem de sucesso dizendo que o cliente foi cadastrado. Então perceba que a gente tem esse fluxo de uma requisição, uma solicitação, que é feita para a nossa API pelo cliente, é processado isso dentro de um servidor, porque a nossa API está dentro de um servidor, inclusive mais a frente você vai ver como fazer o deploy de uma API, ou seja, como colocar sua API em um servidor e disponibilizar essa API para ser utilizada por um front-end, e a nossa API por sua vez, assim que processado, feito a solicitação ali dentro do servidor, devolve a resposta então para o nosso cliente. Tem um outro ponto importante aqui que você precisa saber de uma API, é que sempre uma API vai dar uma resposta, mesmo que essa resposta seja negativa, positiva ou negativa, a API sempre tem que dar uma resposta, porque senão o front-end, a aplicação que está ali no cliente, o cliente vai ficar aguardando, cadê a resposta, cadê a resposta, e vai ficar aguardando, aguardando, aguardando, até estourar o timeout, ou a tolerância de espera da aplicação. Então sempre uma aplicação vai ter que dar uma resposta, quando a gente for desenvolver a nossa API, você vai ver que a gente vai se preocupar com isso, em sempre dar uma resposta, seja negativa ou positiva para quem fez a requisição, tá bom? Bom, para que o cliente se conecte com a nossa API, é necessário uma estratégia de conexão. Então imagina só, aqui a gente tem um cliente que vai fazer a requisição para a nossa API, só que para o cliente saber para quem fazer a requisição, aqui a gente precisa então de uma estratégia de conexão, como conectar o nosso cliente com a nossa API, e aqui vem um outro conceito importante de você conhecer e que a gente vai falar bastante e utilizar bastante na prática, que é rotas, tá? Uma rota é um ponto de extremidade que associa uma URL, e essa URL ela é específica para uma função que manipula requisições HTTP. Em outras palavras, uma rota nada mais é do que o caminho pelo qual o servidor, que é onde a sua API está, responde as requisições HTTP. O HTTP, ele é um protocolo, ele é um padrão de comunicação. Fazendo uma analogia, o português, que é o idioma que nós utilizamos para nos comunicar aqui no Brasil, pode ser visto também como um padrão de comunicação, ou seja, um protocolo. É a maneira como nós nos entendemos, o idioma, a língua que a gente está acostumado e nos entende. O protocolo HTTP é o protocolo utilizado para a comunicação cliente-servidor, que é onde a nossa API está. Então veja que, nesse exemplo, a gente tem aqui uma URL. Veja que a gente tem o HTTPS, dois pontos barra a barra, então o HTTPS é o protocolo, esse S aqui é de security, então do HTTP com o protocolo seguro, quando a gente tem o S, tanto que quando você vai entrar em alguma aplicação web, você vai navegar em algum site, entra em algum site que não tem esse S, que é só HTTP, é muito comum o navegador mostrar para você uma mensagem de, olha, esse site não parece seguro, você tem certeza que deseja realmente continuar aqui, aí aparece ali para você confirmar se você quer realmente seguir, entrar nesse site ou não. Então quando tem um S é porque é seguro, ele tem criptografia, enfim, então quando tem um S é de security, então fica essa curiosidade, mas de uma maneira geral, de um modo geral, a gente se refere ao HTTP, ao protocolo HTTP. Muito bem, então aqui a gente tem o padrão de comunicação que a gente utiliza para a conexão cliente-servidor, aqui a gente tem o endereço de servidor, que é onde de fato o servidor está hospedado, aqui eu coloquei como exemplo meu-servidor.com.br e barra product é a rota, ou seja, qual é o recurso, a funcionalidade que eu quero acessar dentro ali da minha API, então tudo isso é o padrão de comunicação.
