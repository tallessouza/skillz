E o nome disso, existe esse conceito aqui, que é o conceito de Single Thread e é exatamente uma característica do Node. O Node é Single Thread e o que isso significa? Que nunca faz duas coisas exatamente ao mesmo tempo, mas faz uma coisa de cada vez. Olhando aqui agora para o conceito de Single Thread, que é o primeiro conceito aqui que você vai compreender do Node, é que no Node a gente tem uma única Thread principal. Imagina uma Thread como esse encanamento que eu coloquei aqui. Talvez se você já jogou Mario, você vai lembrar de onde é esse encanamento aqui. Muito bem, então pensa que uma Thread é esse encanamento, então a gente tem um único encanamento, um único lugar para as coisas passarem, para as coisas entrarem. Então no Javascript, ou melhor dizendo no Node, no Node há uma única Thread principal que executa o código Javascript. E essa única Thread aqui, ela é responsável por lidar com todas as requisições, executar as funções e gerenciar as operações de I.O. Essa sigla significa Input e Output, ou seja, entrada e saída. E talvez você esteja se perguntando, nossa, mas executar uma coisa de cada vez não seria ruim, ou seria, deixaria lento, enfim, alguma coisa assim do tipo, né? Enfim, isso não prejudicaria o desempenho de uma aplicação Node? Exatamente daí que vem o desempenho. O fato de que essa Thread não fica bloqueada. Então receber uma coisa de cada vez não significa que uma tarefa que eu recebi agora vai bloquear as outras, tá? Então a eficiência nesse sistema de Single Thread que o Node utiliza vem exatamente do fato de que essa Thread não fica bloqueando ou bloqueia a espera das operações de entrada e saída de I.O., Input e Output, quando elas terminam. Então em vez disso, ele vai delegando essas operações e continua processando outras tarefas, tá? Então uma tarefa não bloqueia que está vindo atrás. Esse é outro conceito que é o No Blocking I.O., uma outra característica do Node. Então vamos lá, a gente ficar na mesma página, tá? Primeira característica do Node é Single Thread. Então tem uma única Thread principal que executa o código JavaScript, não faz exatamente tudo ao mesmo tempo, tá? Então uma coisa por vez. E aí a segunda característica é que é No Blocking I.O., ou seja, ele não bloqueia uma tarefa, uma requisição, uma operação, ele não bloqueia as que estão vindo em seguida, tá? Então, por exemplo, a gente tem aqui duas requisições, duas operações. Essa daqui está entrando primeiro, certo? Essa daqui não precisa esperar essa terminar para ela entrar, ela simplesmente vai na sequência ali, tá bom? Muito bem, agora vamos olhar para os pedidos. Imagine que o barista, ele tem uma lista de tarefas para cada pedido e aí cada vez que o barista pega um pedido, ele coloca no topo da lista o pedido e começa a trabalhar nele. Quando ele termina uma tarefa, ele vai lá e risca e pega a próxima, a próxima tarefa que está no topo. E no Node tem alguém que também faz isso e o nome disso é CalStack. A CalStack é outra característica do Node que é como essa lista de tarefa que está ali nas engrenagens do Node. Cada função que precisa ser executada, ela é colocada nessa pilha, ou seja, daí o nome Stack. O Node, ele executa a função no topo da pilha e quando essa função termina, ele vai lá, remove da pilha e segue a vida, passa para a próxima tarefa, tá? Agora a gente vai olhar para uma outra parte importante das engrenagens que é o organizador. Eu vou usar esse termo aqui, o organizador, mas tem um outro nome que a gente já vai ver, tá? Além do barista, a cafeteria tem um organizador. Esse organizador, ele é responsável, você pode chamar de gerente, por exemplo, você pode chamar de garçom, de atendente, enfim. Mas eu vou usar esse nome, o organizador, tá? Ele monitora a lista de tarefas do barista. Por quê? Porque quando o barista está ocupado, o organizador, ele está atento às tarefas que estão terminando. Como, por exemplo, um bolo sendo assado. Não é rápido, leva um tempo ali para assar um bolo, por exemplo. Tem algumas outras atividades que são mais rápidas, por exemplo, pegar uma água. A água já está pronta, é só pegar a garrafinha e entregar. Então, tem tarefas que demandam mais tempo do que outras. E aí, o organizador, ele garante que o barista nunca fique parado e continue trabalhando eficientemente. Daí vem a questão de performance, do porquê essa estrutura, embora uma única thread aqui, mas como ela é muito eficiente. E aí, o nome desse organizador é o Event Loop. O Event Loop é esse organizador, porque ele monitora a call stack e também a fila de eventos, que a gente já vai falar da Event Queue. Se a call stack, ela estiver vazia, ele pega a próxima tarefa da fila de eventos e coloca ela na call stack para ser executada. Agora, vamos dar uma olhada na estrutura inteira aqui e eu já vou te explicar também da fila de eventos. Então, olha só, a gente tem aqui o cliente e entenda como o cliente pode ser uma aplicação web, pode ser uma aplicação mobile, pode ser uma aplicação desktop, enfim. Então, pode ser aqui em diversos contextos, mas de fato o cliente está fazendo uma requisição, está fazendo um pedido. Aqui a gente tem o Event Loop, que é o nosso organizador. Lembra da figura do organizador? E ele é single thread, ou seja, ele vai receber tudo por um único encanamento, digamos assim, e é non-blocking IO, ou seja, a tarefa que está vindo em sequência atrás, então uma tarefa não bloqueia a outra, ou seja, as tarefas não dependem de aguardar uma tarefa terminar para ser executada. Elas vêm entrando dentro dessa única thread, ou seja, single thread, beleza? Quando o Event Loop recebe a tarefa, ele vai ter que tomar uma decisão aqui, porque ele vai olhar bom, essa tarefa aqui, essa requisição, é um pedido complexo? É um pedido simples? Por exemplo, se é um pedido simples como pegar um copo de água, ou pegar uma garrafa de água, algo do tipo, ele pode vir direto para a Calstech e executar essa operação, que é uma operação síncrona, operação fácil, por exemplo, uma soma ali, algum cálculo matemático, alguma coisa que tem execução rápida. Então, uma tarefa síncrona, vai para a Calstech direta ali, já para entrar na pilha de execução, e aí terminou a execução, a operação está completa, devolve a resposta para quem solicitou, que no caso é o cliente. Só que tem tarefas que são um pouco mais complexas, como por exemplo, fazer um bolo, não faz ali no mesmo exato momento, tem que aguardar assar, por exemplo, e ali leva um pouco de tempo. Uma operação assíncrona, por exemplo, seria a leitura de um dado no banco de dados, fazer uma consulta, então leva tempo, então a gente precisa aguardar, por mais rápido que seja, frações de milissegundos, mas ainda assim tem que aguardar. Quando a gente tem essa característica de uma requisição, de um pedido, então ele vem para Event Queue, ou seja, vem aqui para a nossa fila de eventos, é onde ficam as operações assíncronas. E aí, lembra que a gente tinha também, o organizador tem uma outra responsabilidade também. O objetivo do nosso organizador, que é o Event Loop, é nunca deixar o nosso barista aqui sem a tarefa para fazer. Então o objetivo do Event Loop é manter todo o sistema funcionando e o barista ocupado. Então ele vai ficar olhando aqui para a lista de eventos, e quando a calstack estiver disponível, o que ele vai fazer? Aqui ele vai ficar olhando, bom, na lista de eventos tem alguma tarefa que está sendo processada lá, que já está pronta, como por exemplo, o bolo. Ele vai ficar olhando lá, bom, o bolo está assando, está assando. Aí na hora que o bolo estiver pronto, o que ele faz? Ele avisa que está pronto e que já tem aqui na nossa lista de eventos, tarefas que podem ser executadas. Então essa tarefa pronta vem aqui para a calstack para ser executada, e aí então a operação completa devolve a resposta para o cliente.
