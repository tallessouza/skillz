Autenticação é o processo de verificar a identidade de um usuário, ou seja, identificar quem é o usuário. É como provar que aquela pessoa é realmente quem ela diz ser. Então imagina só, quando você vai fazer um cadastro em alguma loja ou em algum lugar, o que geralmente se pede é o seu documento. O seu documento é uma maneira de provar que você é quem você está dizendo ser. Então é um processo de autenticação, de identificar quem você é. E aí, aqui tem um exemplo de um fluxo de autenticação, que é quando o usuário faz o login na nossa aplicação ou em uma aplicação. Quando o usuário faz o login, ele fornece as credenciais, por exemplo, e-mail e senha. Quando você vai entrar no Instagram, no Facebook, no LinkedIn ou em qualquer outra plataforma ou site, você vai fazer o login, provavelmente vai te pedir o e-mail ou senha, ou o usuário e senha. Então, a partir daí, você está fazendo um fluxo de autenticação. Você está dizendo quem você é e aí a plataforma, o site, vai identificar se você é realmente quem você diz ser. Então, se as credenciais forem válidas, o usuário é considerado ou será considerado autenticado. Então, imagina só, o cliente solicita a autenticação com e-mail e senha para o back-end. O back-end recebe o e-mail e a senha e vai verificar se esse usuário existe, por exemplo, no banco. Por exemplo, esse e-mail é um e-mail que existe? Tem algum usuário no banco de dados com esse e-mail? Se tem com esse e-mail, a senha confere? Realmente é a mesma senha? Então, esse processo de verificar se o usuário existe. Se o usuário não é válido, ou seja, o usuário não existe porque ou não tem um usuário com esse e-mail cadastrado ou até tem um usuário com esse e-mail, porém a senha não bate, não é uma senha correta, então a gente retorna uma mensagem, por exemplo, de e-mail ou senha inválido. Agora, se o e-mail realmente bate, existe um usuário com esse e-mail e a senha realmente confere e-mail e senha estão incorretos, então significa que esse usuário é realmente quem diz ser e a gente retorna para o usuário um token de autenticação. A partir desse momento, a gente está retornando para o usuário como se fosse um crachá, digamos assim, que vai dar acesso para a nossa aplicação. E aí a gente passa agora para o fluxo de autorização. A autorização define o que um usuário que está autenticado pode ou não fazer dentro do sistema. Então, quando o usuário faz a autenticação, ele não recebe o token, esse token vai ser como um crachá. Depois de autenticado, ele consegue mostrar para o sistema, para a nossa aplicação, para o back-end, quem ele é e o sistema verifica quais permissões esse usuário tem. Olha só, vamos agora adicionar aqui no nosso fluxo de autorização, vamos agora olhar para o nosso fluxo de autorização, ou seja, um usuário ele pode ser autenticado no sistema, mas ainda assim não significa que ele tem permissão para fazer o que ele quiser, por exemplo, para ele acessar um painel administrativo ou excluir dados de alguma tabela. Não, a autorização é quem vai determinar esse nível de acesso com base no papel, ou na rule, ou na permissão que esse usuário tem dentro da nossa aplicação. Então, olha só, digamos que o cliente, e aqui o cliente imagina pode ser, de fato, uma pessoa, pode ser uma aplicação front-end web, mobile, enfim. Então, a gente tem uma requisição e essa requisição tem um token. Inclusive, você vai aprender a inserir um token na requisição e esse token tem ali as informações desse usuário, quem é o usuário que está fazendo essa requisição. E o middler que a gente vai criar vai identificar quem é esse usuário e, identificando esse usuário, a gente sabe se ele tem autorização para fazer o que ele está pedindo para fazer. Ou seja, o usuário pode fazer isso dentro do banco? Se sim, segue o fluxo. Dentro do banco ou dentro da nossa aplicação? Se sim, segue o fluxo e deixa, por exemplo, o usuário cadastrar um produto. Mas, digamos que esse usuário não pode cadastrar um produto, então, se esse usuário pode cadastrar um produto, beleza, ele vai seguir com o fluxo e tem a resposta. Se ele não pode, então a gente para por aqui, nem continua com a solicitação do usuário e devolve para ele um não autorizado. Dá até para usar o status code 401 de não autorizado. Então, a gente barra o usuário. Olha, o que você está pedindo aqui para fazer, você não tem autorização para fazer. Assim como em uma empresa tem aqueles setores que, por exemplo, um banco. A gente pode definir níveis de acesso, definir quem pode fazer o que, que tipo de perfil, de permissão pode fazer o que dentro da nossa aplicação. Para resumir, a autenticação é quem você é, é o processo de identificar o usuário e autorização é o que o usuário pode fazer, é o processo de dar permissão e verificar se o usuário pode ou não fazer aquilo.
