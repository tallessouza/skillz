Bom, outro princípio é esse de manipulação de recursos através de representações, em uma arquitetura REST os recursos eles são manipulados ou modificados, não diretamente, mas por meio de representações, o que a gente usa, por exemplo, para representar os dados que trafegam da requisição, então o que a gente envia na requisição ou obtém ali na resposta é no formato JSON, o JSON é um formato para a gente representar dados através de objetos, mas não é da forma que necessariamente ele vai ser salvo, por exemplo, dentro de um banco relacional, mais à frente você vai compreender ali e vai entender essa parte de banco de dados, mas um banco de dados relacional vai guardar os dados, por exemplo, no formato de tabelas e linhas, e o banco de dados está lá dentro do servidor, então o servidor, a nossa API, ela vai receber os dados, por exemplo, no formato JSON, que é a maneira como a gente representa esses dados, mas a nossa API vai manipular aqueles dados para poder armazenar dentro do banco que está ali no servidor, do jeito que o banco de dados armazena, então a gente utiliza aqui uma estratégia para representar esses dados. Outra característica, outro princípio, é o de sistema em camadas, que ajuda a gente a organizar e separar as responsabilidades para a gente alcançar uma arquitetura que seja eficaz e também de fácil manutenção, e por fim, um outro princípio aqui de cachear, de cache, que basicamente incentiva que o nosso front-end reutilize os recursos que já foram obtidos, em vez de ficar fazendo novas requisições repetidamente, pedindo as mesmas informações do servidor. Então esses são princípios que fazem de uma API, uma API RESTful, e basicamente uma API, ela é considerada uma API ou API RESTful, se ela cumpre as diretrizes REST. Muito legal, né?
