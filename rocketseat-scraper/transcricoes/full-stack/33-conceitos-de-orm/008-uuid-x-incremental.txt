WEBVTT Nessa aula eu vou explicar pra você a diferença entre o ID do tipo incremental, que é esse daqui, e o ID do tipo UUID, que é um tipo de ID que a gente vai começar a utilizar na nossa aplicação. Vamos relembrar aqui o incremental, né. O incremental, ele gera o ID de 1 em 1. Então, o primeiro ID, por exemplo, vai ser 1. Quando eu registrar um novo usuário, ele vai ser o 2. Quando eu registrar de novo, vai ser o 3. Então, baseado no último ID gerado, ele gera o próximo, ou seja, de 1 em 1, em incremental. Então, basicamente, essa é a ideia central de um ID do tipo incremental. E aí tem requisições, que são requisições que a gente precisa passar, por exemplo, o ID do que a gente precisa acessar especificamente. Digamos que aqui a gente queira, por exemplo, acessar o ID, o usuário com o ID 2, né. Então, é normal a gente passar esse ID aqui na requisição, aqui como um parâmetro, né, da requisição. E isso daqui é o que a gente chama também de ID público, que é aquele ID também que acaba ficando exposto na requisição. E aqui, como é um ID do tipo incremental, é fácil a gente chutar, ou um usuário mal-intencionado chutar um ID de usuário. Porque o incremental, ele é mais fácil, sabendo que ele vai gerando ali de 1 em 1, de 2 em 2, enfim, né, numa sequência. E aí que entra o UUID. Olha só, o padrão do tipo UUID, ele é uma combinação de números e letras bem grande, que é o que a gente também chama de hash. Ele separa esses grupos por traço, tá vendo? Tem uns traços aqui que separam esse grupo. E esses grupos, eles têm uma complexidade maior para gerar esses valores. Então, aqui são milissegundos, aqui são dois bits, variantes, enfim, você não precisa se preocupar em como ele gera esse código, não é esse o ponto aqui. O ponto é você entender que ele é um valor gerado com uma complexidade maior. Então, a gente tem aqui um ID de um tamanho bem maior, um conteúdo aqui também maior e, portanto, é mais difícil um usuário mal-intencionado ficar chutando um ID aleatório. Porque ele é baseado em um padrão gerado e aí, se o ID não é válido, então ele não vai conseguir encontrar e ficar palpitando aqui, chutando um ID de usuário, por exemplo, numa requisição que tem um ID sendo passado publicamente. Então, por isso, é legal você conhecer, claro, o incremental é muito bom para ser utilizado. Você pode utilizar ele para esses IDs que são mais privados, que vão ser utilizados ali mais dentro do banco e para IDs com essa característica de serem mais, de potencialmente utilizar ele como um ID público, que vai ser utilizado, por exemplo, ali numa requisição, como esse exemplo que eu dei. Então, a gente pode utilizar um UUID que tem uma complexidade bem maior, que tem uma validação e, inclusive, torna mais difícil um usuário ou alguém mal-intencionado de ficar tentando passar aqui um ID nesse padrão UUID de forma aleatória. Então, agora que você sabe a diferença, então basicamente essa é a diferença. Então, a gente tem no UUID esse padrão aqui para gerar o ID e também a gente vai utilizar um método para gerar um ID do tipo UUID automaticamente.
