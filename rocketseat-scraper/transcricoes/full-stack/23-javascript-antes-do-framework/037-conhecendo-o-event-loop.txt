WEBVTT Nessa aula você vai compreender o que é o Event Loop e vai te ajudar a desbloquear sua mente para compreender como as coisas funcionam no JavaScript. E claro, a gente vai ver também na prática, mas vamos focar em entender primeiro o que é o Event Loop. Bom, primeiro vamos dar uma olhada nas características do próprio JavaScript. Primeiro, o JavaScript é Single Thread, isso significa que o JavaScript executa uma coisa por vez. Imagina que você vai em algum lugar, você tem que esperar, e só tem uma única fila, só tem uma pessoa atendendo nesse lugar, e tem uma fila lá, e você tem que esperar chegar a sua vez. Então, uma única fila, uma Single Thread. Outra característica do JavaScript é que ele é No Blocking, então ele não trava o contexto da execução. Por exemplo, quando a gente vai usar uma função assíncrona, se você não pede para o JavaScript aguardar a promise ser resolvida, ele vai seguir a vida e vai continuar a execução dele. O que leva a gente para outra característica de ser assíncrona, que é essa característica de No Blocking, então a gente precisa utilizar um paradigma assíncrono para poder lidar com a execução das coisas, o que leva a gente para a última característica de concorrência. As tarefas que executam assincronamente, ou de forma assíncrona, elas concorrem umas com as outras pelo processamento da tarefa em si, e o próprio JavaScript precisa então gerenciar essa execução. E é aí que entra esse conceito de concorrência e o próprio Event Loop. O JavaScript possui um modelo de concorrência que é baseado no Event Loop, e o Event Loop é responsável por gerenciar a execução do código assíncrono e eventos em uma única thread. E é isso que permite que o JavaScript seja não bloqueante, então esse é o papel do Event Loop. A gente tem várias tarefas concorrendo para ser processadas e o Event Loop é quem vai gerenciar quem vai ser executada. Vamos dar uma olhada nesse diagrama que a gente tem aqui, as engrenagens de como funcionam as coisas. A gente tem aqui a call stack, e a call stack é a fila de chamada, ela vai armazenar as chamadas de funções em execução. Quando uma função é chamada, ela vem aqui para o topo dessa pilha. A Web API são recursos que vão ser executados no navegador, vai ser executado em algum outro lugar, em alguma Web API. Ela passa pela call stack porque tudo passa pela call stack. Algumas tarefas vão ser executadas nela mesma, mas outras só passam mesmo, que é o caso da Web API, vão ser executadas em outro lugar. A callback UI guarda ali para a gente, ela é uma fila também, e ela armazena as callbacks e eventos que estão sendo ali aguardados para ser processados. Estão aguardando, estão esperando um clique da vida, um download e assim por diante. E aí tudo isso acontece em memória. Onde entra o Event Loop aqui? O Event Loop fica verificando constantemente, constantemente ele fica olhando para essas duas filas, a call stack, a nossa fila de chamada aqui das funções em execução e a callback UI, que são aquelas que estão aguardando para serem processadas. Então se eu tenho aqui, por exemplo, uma aguardando, ele vai pegar ela, tirar ela daqui e vai colocar ela aqui no topo da call stack para ela então ser chamada para execução. Então ele sempre vai gerenciando de forma constante. Ele tem um critério de prioridade de execução que a gente já vai falar daqui a pouco. Bom, tudo passa pela call stack, como a gente viu. Algumas tarefas vão ser resolvidas na própria call stack mesmo e outras só passa por ela e não vai fazer nada ali e vai ser resolvido em alguma outra API. O Event Loop é quem fica então checando de forma constante a call stack e a callback UI para fazer a execução. E como eu mencionei no slide anterior, existem prioridades e aí a gente tem basicamente dois tipos de tarefas na fila de callback. E aí já falo das prioridades de acordo com os tipos de tarefas. A gente tem Microtasks e Macrotasks. Vamos falar primeiro das Microtasks. Elas são tarefas de alta prioridade e elas vão ser executadas antes das Macrotasks. Aqui entram, por exemplo, temporizadores e promises. As Macrotasks são tarefas de menor prioridade como callbacks, de eventos, setTimeout e setInterval e aí elas vão ter prioridades menores. Aqui tem um resumo, tá? Acompanha comigo. Primeiro, a gente tem a execução de código. O código síncrono vai ser executado de cima para baixo, empilhando então e desempilhando funções conforme necessário. Próxima etapa, eventos assíncronos. Quando ocorrem eventos assíncronos, como uma requisição concluída, o callback correspondente é enfileirado lá na fila de callback. O event loop vai verificar a pilha de chamada e a fila de callback. Se a pilha estiver vazia, ele move um callback da fila de callback para a pilha de chamada para poder executar a função. E aí é onde entram as Microtasks. Antes de verificar novamente a fila de callback, o event loop sempre vai ver se tem alguma Microtask na fila, pendente, para trazer ela para a prioridade de execução. Esse é o resumo do que eu expliquei anteriormente e agora então você conhece o event loop.
