E uma tabela em um banco de dados relacional tem uma estrutura muito parecida. Além de organizar os dados em tabelas, nessa estrutura de linhas e colunas, o banco de dados relacional tem esse nome relacional porque uma tabela pode se relacionar com outras tabelas. E a base desse relacionamento são chaves comuns, chaves primárias e estrangeiras, mas não precisa se preocupar com isso agora. Então, continuando aqui, agora vamos falar de cada parte dessa estrutura de um banco de dados relacional, começando pela tabela que é a base. Ela é a base de um banco de dados relacional, do armazenamento de um banco de dados relacional. Bom, uma tabela é uma estrutura de armazenamento fundamental em um banco de dados relacional, como eu mencionei, e uma tabela sempre vai guardar dados de um assunto. Não é recomendado você colocar dentro da mesma tabela diversos assuntos. Imagina só, uma tabela que guarda dados de produto, dados de cliente, dados de usuário, vira uma bagunça. O ideal e o recomendado é cada tabela guardar dados de um assunto. Por exemplo, temos aqui uma tabela de produtos. Na tabela de produtos, vamos guardar informações, ou melhor dizendo, dados de produtos. Temos uma tabela de usuário, vamos guardar dados de usuários, uma tabela para cliente, uma tabela para categoria. Então, perceba que cada tabela tem um assunto e, portanto, vamos seguir a boa prática de guardar sempre, separado, dados por assunto. Então, isso organiza também a nossa base de dados, o nosso banco de dados. Também é bem comum escrever o nome das tabelas no plural, por exemplo, produtos, porque uma tabela guarda vários produtos, registros, diversos registros, vários registros de produtos. Por isso, é muito comum utilizar o nome de tabelas no plural. Bom, agora vamos olhar dentro de uma tabela. Qual é a estrutura de uma tabela? Bom, uma tabela é composta por linhas, que também são chamadas de registros, que a gente visualiza aqui na horizontal, também conhecidos por tuplas. Então, é bem comum também esse nome tuplas, que significa registro. São as linhas aqui de uma tabela. Então, veja que a gente tem um registro, está vendo? Aqui a gente tem um segundo registro e as colunas também são chamadas de campos ou atributos. As colunas têm a função de separar os tipos de dados. Então, por exemplo, veja que a gente tem uma coluna para o ID, abreviação aqui de identifier, o identificador, e aqui a gente tem uma coluna, então, para o ID, esse é o nome da coluna, para números inteiros. Então, veja que na coluna ID só temos o tipo de dado inteiro, ou seja, número inteiro. 1, 2 e poderia ter outros aqui. Na coluna title, para título, temos aqui um tipo de dado que é texto. Aqui na nota temos ponto flutuante, ou com casa decimal, 9.8, 9.5. Então, a coluna, ela determina qual é o tipo de dado. Cada coluna vai guardar dados do mesmo tipo. Então, tipos numéricos, tipos de texto, e assim por diante. E a linha compõe um registro. Então, nessa linha selecionada, temos o ID 1, esse é o título, essa é a nota e essa é a data quando foi criada. Então, perceba que na horizontal a gente tem um registro e na vertical, nas colunas, a gente tem os dados separados por tipos. Agora, vamos ver outra característica de uma tabela de um banco de dados relacional, que são as chaves primárias. É muito comum utilizarmos chaves primárias nos identificadores ou no ID. A primary key, ou chaves primárias, elas identificam exclusivamente cada registro em uma tabela, e o objetivo dela é garantir que cada registro seja único dentro da tabela. Então, esse identificador, ele é importante para quando a gente quiser deletar um registro específico, ou quando a gente quiser atualizar um registro específico, a gente precisa ter uma maneira de diferenciar um registro do outro, é aí que entram, então, os identificadores. É muito comum a tabela ter como identificador um ID. Então, veja que aqui a gente tem um registro 1, aqui a gente tem um registro de ID 2, então é bem comum utilizar o ID. Só que, só utilizar uma coluna com o nome de ID não garante que esse registro seja único. Para isso, existe a chave primária. Quando a gente coloca que uma coluna é chave primária, o próprio banco de dados vai garantir que esse registro seja único. Isso significa que, se eu tentar cadastrar um registro com um ID que já existe, ele não vai deixar, ele vai restringir para que isso não aconteça, e essa é a consistência e a segurança que a gente viu em aulas aí, na aula passada, que é uma das características também de banco de dados. Uma tabela é composta por colunas, então tem coluna A, coluna B aqui, por exemplo, mas poderia ter uma coluna com ID, com NAME, PRICE, e por aí vai. E, além das colunas, temos os registros, e essa estrutura se repete para as outras tabelas também. Então, essa é a maneira como o banco de dados relacional organiza e estrutura os dados.
