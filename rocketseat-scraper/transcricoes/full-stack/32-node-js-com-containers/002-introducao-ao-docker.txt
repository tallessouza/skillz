WEBVTT Nessa aula vamos fazer uma introdução ao Docker para você compreender o que é o Docker e também os conceitos do Docker. Começando então pela definição o que é o Docker. O Docker é um sistema de virtualização não convencional e eu vou te explicar o porquê. Começando então a entender a virtualização convencional. Aqui do lado tem uma ilustração, tá? Bom, em virtualizações convencionais a gente tem um software que é instalado na máquina host. Veja que aqui a gente tem a máquina host, por exemplo, a nossa própria máquina física e na nossa própria máquina a gente instala um software de virtualização. Para que serve esse software de virtualização? Para gerenciar máquinas virtuais. Grandes exemplos de software de virtualização é o VirtualBox, VMware, Parallels, entre outros, tá? E com esse software de virtualização a gente consegue criar máquinas virtuais e para cada máquina virtual a gente tem uma instalação completa do sistema operacional que a gente quer virtualizar, além de ter o próprio hardware virtualizado. Por exemplo, digamos que aqui eu quero criar uma máquina virtual com Windows. Ali eu quero criar uma máquina virtual com Linux. Aqui eu posso criar uma máquina virtual com MacOS e aqui as outras podem repetir. Eu posso ter outra com Windows também ou com Linux e assim por diante. E para cada máquina virtual a gente tem uma instalação completa do sistema operacional, certo? Com o Docker a gente usa uma abordagem diferente. A gente utiliza o conceito de container. Agora vamos entender o que são os containers. Os containers, eles proporcionam um ambiente isolado com os recursos que a sua aplicação precisa para funcionar. Como, por exemplo, o código da aplicação, as dependências e bibliotecas necessárias para executar a nossa aplicação e também fazer a nossa aplicação funcionar. Imagina um navio cargueiro que carrega um monte de container. Esse navio ele é o nosso host e cada container é um ambiente completamente isolado. Se um container falha, como ele é um componente ou um container isolado, ele não afeta o restante da operação. Digamos que está em alto mar e aí o mar está muito agitado e por algum motivo um container caiu em alto mar. Os demais containers não são afetados porque eles são ambientes isolados. Cada container só tem conhecimento do conteúdo ali, do que está dentro do seu próprio container. Um container não consegue visualizar o que tem dentro do outro container. Então esse é o princípio de isolamento. Continuando aqui, agora vamos comparar a virtualização com os containers utilizados pelo Docker. Na virtualização a gente tem a infraestrutura, por exemplo, aqui a máquina física, o host, por exemplo, e a gente pode ter cada VM ou cada máquina virtual, virtual machine, com um ambiente diferente, um sistema operacional diferente e aí sempre a gente tem uma nova instalação, um novo sistema operacional. Com os containers a gente tem o mesmo sistema operacional sendo utilizado e compartilhado pelos containers, ou seja, utilizando aqui o Linux. Então para trabalhar com containers nós vamos utilizar o Linux e isso faz dos containers serem muito enxutos, mas muito enxutos comparados às máquinas virtuais. Cada máquina virtual é um novo sistema operacional, uma nova instalação. No container a gente tem inclusive o mesmo sistema operacional sendo compartilhado e cada container seguindo o princípio de ser isolado. Muito bem, seguindo aqui no aspecto do isolamento, vamos agora focar nesse princípio aqui do container para você entender como isso é possível. Veja que aqui na ilustração a gente tem o host e aí a gente tem o container 1, container 2, container 3. O host é onde a gente está executando então os nossos containers e veja que aqui a gente tem o kernel e aí a gente tem os cgroups. Vou explicar cada um deles. Começando aqui pelo kernel. O kernel é o coração de um sistema operacional. No caso aqui dos containers a gente vai utilizar Linux. Os containers com docker utilizam Linux, que faz a ponte entre o software e o hardware e controla inclusive processos, memória, dispositivos e chamadas do sistema operacional. Então o kernel faz essa interface entre software e hardware do computador. O hardware é ali as peças, memória, o processador, então a parte física que conversa com o software, com os processos ali que a nossa aplicação utiliza. Então o kernel é responsável por fazer esse meio de campo. O cgroups é uma funcionalidade que controla e veja que ele roda a nível do kernel. Então ele está próximo ali, fazendo essa interface entre software e hardware e ele controla e limita a alocação de recursos, como por exemplo CPU, o uso de memória e assim por diante. E qual é o objetivo disso? O objetivo é não deixar que um container monopolize os recursos do host, ou seja, imagina que um container pega toda a memória só para ele e os outros ficam com pouca memória. Não, então o cgroup está aqui para manter equilibrado esse ambiente, para que todos os containers tenham recursos necessários para funcionarem bem. E os namespaces, eles isolam os recursos, ou seja, com o namespace ele é responsável por fazer um container só enxergar os seus próprios processos, seus próprios arquivos, por isso que um container não vai conseguir visualizar o conteúdo de outro container. Muito bem, agora que você entendeu esse princípio do isolamento e como ele funciona, por que então utilizar o docker? O docker ele resolve o famoso problema na minha máquina funciona. Bom, com o docker a gente tem um container que tem a nossa aplicação e esse container ele pode ser levado inteiro para outro ambiente. Com isso você não precisa se preocupar um pré-requisitos instalados no outro ambiente, versão de sistema operacional, permissões e por aí vai. Então dessa forma a gente minimiza ou minimizamos muito a divergência entre os ambientes. Então você constrói um ambiente que é padronizado para rodar sua aplicação em cima de uma imagem e consegue levar esse seu container para executar em outros lugares. Então você consegue padronizar o ambiente em que a sua aplicação precisa e irá rodar. E aí você consegue levar esse container, por exemplo, para rodar no servidor. Hoje ele está rodando aqui na sua máquina, mas depois na hora de fazer o deploy você pode, por exemplo, levar ele para um servidor e assim por diante. Bom, outros conceitos aqui de docker para a gente finalizar nossa aula. Quando vamos trabalhar com docker e a gente vai passar por cada um desses aqui, você vai ver que nós vamos criar um docker file, ou seja, um arquivo que contém todas as informações necessárias para gerar a nossa imagem docker. A imagem, por sua vez, ela contém as informações de um ambiente com tudo que a nossa aplicação precisa para executar. Por exemplo, código, dependências, bibliotecas necessárias e assim por diante. Inclusive nós temos o docker hub, que é um lugar onde a gente tem imagens disponíveis para a gente utilizar. E aí tem as imagens oficiais, imagens criadas e compartilhadas pela comunidade e assim por diante. E temos o container, que é a instância de uma imagem em execução, ou seja, é o ambiente de uma imagem executando. E aí a gente tem a nossa máquina hospedeira, que é o nosso host, que pode ser uma máquina virtual rodando em uma máquina física como um servidor ou diretamente na máquina física como o nosso computador. Então, nessa aula você compreendeu o que é o docker e os conceitos de docker. Agora vamos começar a utilizar o docker na prática.
