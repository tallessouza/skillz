Os princípios SOLID são cinco princípios que vão reger boas práticas de se utilizar em projetos uma linguagem de programação que tenha vínculo com o paradigma de orientação a objetos. Os princípios SOLID foram feitos para te mostrar como utilizar uma linguagem de programação vinculada ao paradigma de orientação a objetos, que seria o nosso caso com a linguagem Kotlin. Essas diretrizes vão te mostrar como você consegue escrever códigos que vão ser mais modulares, mais flexíveis para você poder trabalhar e também mais fáceis de manter ao longo do tempo, durante o crescimento do seu projeto. Esses princípios foram descritos inicialmente por Robert C. Martin, o famoso Uncle Bob, e por mais que eles tenham sido criados especificamente para orientação a objetos, eles também podem ser aplicados em outros paradigmas, como por exemplo programação funcional e conceitos de arquitetura de software no geral. E agora nós vamos analisar um a um desses princípios, mostrando como você deve seguir cada um deles. Começando pela letra S, o Single Responsibility Principle, o SRP, que traduzindo para o português seria o princípio da responsabilidade única. E ele é muito simples. Basicamente, todo bloco de código que você constrói deve ter apenas uma única responsabilidade. Um único propósito de execução. Ou como o próprio Uncle Bob diz, uma classe deve ter apenas um motivo para mudar. Um exemplo seria a gente separar a lógica de gerenciamento de um banco de dados, por exemplo, como que a gente construiu, utilizando a biblioteca RUN em uma classe de data source, de fonte de dados. Ao invés da gente misturar essa lógica de gerenciamento com a lógica de construção do banco. Dessa forma, o arquivo do qual estamos gerenciando o banco de dados tem uma única responsabilidade, que seria o do próprio gerenciamento do banco. Já a classe responsável por construir a instância do banco de dados possui apenas a responsabilidade de construção. Então os motivos de mudança de cada uma delas estão separados. Ao criar uma classe desse tipo, o nosso gerente de usuários, ele passa a ter duas responsabilidades, a de registro e a de envio de e-mails, ou seja, dois motivos para mudar quando nós quisermos alterar a lógica de registro de usuário e quando nós quisermos também alterar a lógica de envio de e-mails. Essa seria uma implementação que quebra o princípio de responsabilidade única. Certo? Ok, agora seguindo para o próximo princípio, nós temos a letra O, que representa em inglês o princípio Open Closed, ou seja, o princípio aberto fechado. O que esse princípio entrega para nós de regras? Ele implica que um código, ao ser construído, deve estar aberto para extensão e fechado para modificação. Isso representa que você poderá adicionar sempre novas funcionalidades a um código que já existe no seu projeto. Certo? Porém, o código já existente, ele não deve ser alterado. Para que essas novas funcionalidades possam existir.
