WEBVTT E agora que chegamos ao fim do nosso projeto do aplicativo RocketA, nós vamos tirar um tempo para a gente rever os conceitos de arquitetura que nós vimos no início desse módulo sendo aplicados de fato em um projeto. Começando aqui, primeiro pela separação de conceitos. A gente já percebe na árvore principal do nosso projeto que nós definimos muito bem as responsabilidades que cada pacote iria ter, tudo seguindo a base de pacotes recomendada oficialmente pelo Android. Então nós vamos ter uma separação de conceitos relacionados às regras de negócio, as lógicas da própria aplicação na camada de domínio e todas as dependências de UI e toda a parte que referencia as UIs estão centralizadas e organizadas em pacotes condizentes com suas nomenclaturas. Além disso, nós construímos um fluxo baseado em arquitetura limpa para a gente conseguir alcançar um padrão preciso de como construir uma funcionalidade. Então aqui nessa imagem que a gente viu na aula sobre arquitetura limpa, a gente consegue perceber cada uma das etapas sendo mapeada através de pacotes aqui no nosso projeto. Certo? Então analisando a camada mais interna, nós temos API e database sendo contabilizados, além disso a gente tem inclusive preferences como sendo uma camada adicional aqui do framework Para entrega de dados Nós temos as implementações dos data sources Tanto interface quanto implementação estão sendo definidas aqui no nosso módulo de data Certo? Inserimos também a implementação de repositório aqui. Agora, pulando para a camada de domínio, nós já vemos a interface do repositório e também os modelos de domínio e os use cases sendo entregues aqui nessa parte também. Por isso na camada de apresentação, que seria essa camada de UI, nós temos tanto ViewModels, quanto também as nossas Activities, Fragments e demais classes que são caracterizadas por ter um vínculo direto com UI. Como por exemplo, os nossos eventos, states e funções de extensão e adapters que a gente está utilizando apenas aqui no contexto da nossa view Certo? Inclusive falando de eventos e states Aplicamos uma mistura de MVVM com MVI aqui no nosso projeto Certo? Os eventos que a gente está tratando Lá nos nossos métodos OnEvent Eu vou abrir aqui AI ChatViewModel e AI Chat, para a gente pegar de exemplo, vou fazer um split aqui para a gente ver ambos os códigos, fechar aqui essa imagem, demonstram que a única função que a gente expõe do nosso View Model seria a função OnEvent, que processa os eventos e executa funções internas aqui do ViewModel para conversar diretamente com a nossa camada de regra de negócio, através, no caso, dos nossos use cases. Porém, na hora de expor os dados observáveis, a gente está utilizando vários state flows. Certo? A gente não tem um único UI state sendo entregue. A gente apenas fez esse formato na nossa tela de Welcome onde a gente criou de fato um Welcome UI State porém ele só contém uma única propriedade como havia sido explicado quando a gente tem múltiplas propriedades a serem mostradas dentro de um único UI State ele poderá gerar um contexto dentro das views criadas em XML de atualizações desnecessárias de valores que são iguais dentro das propriedades dos UI States. Por isso a gente manteve essa somatória de MVVM com MVI nesse nosso formato de app. Esse UI State a gente acabou não utilizando, porém é um formato de dado que a gente consegue trabalhar dentro dos ViewModels, caso a gente queira observar dentro das nossas Views diferentes estados que representem sucesso, erro e loading. Em diferentes instantes em que o usuário está solicitando algo para a camada de domínio. E além disso, a gente percebe também que nós implementamos, através das construções de interfaces e implementações, por exemplo, de repositórios e data source, a inversão de dependências. Certo? Nossas classes sempre sempre dependem Das interfaces E nunca das implementações E cada uma delas Possui apenas uma única responsabilidade Nós surgimos ao máximo Da abrangência de múltiplos contextos Dentro de uma única classe ou método Além disso nós tivemos a oportunidade também De aplicar a injeção de dependências Utando tanto o Dagger quanto o Hilt de forma apartada. Aqui no nosso pacote de CoreDI, você aprendeu a construir os códigos de injeção utilizando ambas as bibliotecas e nós tentamos ao máximo simplificar os nossos códigos através de funções de extensão, deixando eles simples, diretos e o mais estúpidos possíveis, aplicando ao mesmo tempo o Kiss para mantermos tudo simples e também o Dry para não nos repetirmos ao utilizar nossos códigos. Dessa forma forma nós mantivemos sempre os nossos fragments limpos contendo apenas lógicas diretas sobre as execuções que de fato a gente gostaria de rodar em cada um dos momentos de interação com o usuário tivemos também convenções de nomenclatura sendo usadas em todo momento para mantermos sempre os mesmos padrões para classes que possuíam coisas em comum. Por exemplo, os nossos view models tendo sempre funções com o mesmo nome de onInvent para processamento de eventos. Os nossos recursos sempre passando IDs e nomenclaturas de arquivos seguindo o mesmo padrão além de outras classes que a gente foi definindo em todos os nossos pacotes para garantirmos o melhor entendimento possível de quem leu de fato nosso código e é isso esses foram alguns highlights dos conceitos de arquitetura que a gente aplicou no projeto e agora a gente parte para a conclusão vendo uma forma de se modularizar o aplicativo RocketEar e garantir uma melhor escalabilidade, manutenção e testabilidade para o futuro desse projeto.
