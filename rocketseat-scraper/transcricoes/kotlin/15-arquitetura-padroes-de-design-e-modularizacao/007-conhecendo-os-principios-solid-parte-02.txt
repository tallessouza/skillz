Partindo agora para o nosso penúltimo princípio, solid, nós temos a letra I de Interface Segregation Principle, a segregação de interfaces. A segregação de interfaces é um dos princípios mais simples de se entender. E ele dita que uma interface nunca deve forçar as classes que a estendem de implementar métodos que essas classes não querem usar ou não devam utilizar. É muito melhor você ter dentro do seu projeto várias interfaces que são específicas do que uma única interface que é muito grande e que vai exigir várias implementações vazias para poder apenas concluir a implementação daquele contrato que a interface rege. Então sempre que você cair nesse cenário, pense em uma forma de separar as suas interfaces, certo? Um exemplo bem fácil e intuitivo para a gente entender, é dentro de um projeto nós criarmos uma interface animal com os métodos que representam animais no geral que poderia ser voar, nadar, correr porém ao implementar essa interface animal para, por exemplo, um leão, exigiria, dentro do seu código, uma implementação vazia para o método voar. E isso quebra o princípio de segregação de interfaces. Então, para corrigir isso, a ideia é criar interfaces que sejam separadas, visando mapear os comportamentos dos animais e que complementam a implementação da interface animal. Nesse exemplo, poderia ser utilizado interfaces nadador, voador e corredor e cada uma delas iriam conter cada um dos métodos específicos desses comportamentos de animais.
