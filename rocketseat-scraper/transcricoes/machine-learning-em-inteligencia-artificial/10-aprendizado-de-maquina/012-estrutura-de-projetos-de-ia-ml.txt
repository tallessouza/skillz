E é muito importante a gente ter algumas formas de estruturação, de metodologia, para que a gente possa padronizar o processo, para que a gente possa ter uma forma de reproduzir o processo. Tem um cientista de dados ou um especialista que quer entender como que aquele projeto foi feito. Quando esse projeto está numa estrutura já conhecida, fica fácil de ele conseguir reproduzir, entender. que ponto começou, que ponto que está, que ponto que ainda tem que seguir. Então tudo isso, essa estrutura toda facilita o trabalho do cientista, facilita o trabalho de quem está atuando no projeto. Além disso, fica fácil também de documentar, ou seja, garantir que haja uma documentação mínima sobre o projeto e também facilita também as interações. O projeto de Machine Learning de Inteligência Artificial é um projeto de natureza experimental, interativa. Muitas vezes você faz um modelo aquele modelo está, ele é satisfeito até o momento, ele gera um resultado, mas você está ali muitas vezes revisitando aquele modelo, revisitando as métricas daquele modelo para propor coisas melhores. Então eu vou trazer aqui algumas das estruturas ou técnicas ou frameworks usados no mercado hoje para esses projetos de AI Machine Learning. O primeiro deles é o, vamos dizer, o mais clássico, mas que ainda é, e muito, acho que talvez o mais usado ainda hoje em dia, que é o chamado CRISP-DM, que é um acrônimo para Cross-Industry Standard Process for Data Mining. Ele foi criado em 1996 e ele acabou se tornando na metodologia o framework mais difundido em ciência de dados para uso em machine learning ou inteligência artificial. E a estrutura do CrispDM é cíclica. Então é possível que você, depois de passar por uma etapa, você tenha que retornar para uma etapa anterior para fazer algum tipo de refinamento. Então como que ele funciona? Mostrando nesse desenho aqui. Eu parto de uma etapa inicial de entendimento do negócio, mais precisamente entendimento do problema que eu quero resolver. Baseado nesse entendimento de problema, eu passo a entender os dados que eu tenho disponíveis que podem resolver esse problema. Então, baseado nesse entendimento dos dados, eu posso seguir em frente, mas eu posso fazer novas perguntas para o negócio. Olha, eu entendi esse dado ou não entendi esse dado, em que este dado, essa informação poderia me ajudar a resolver o problema? Então, já tenho aqui uma interação. Posso ficar aqui tentando entender o negócio e, com base nos dados, fazer novas perguntas. Tendo esse entendimento, eu passo para uma etapa de preparação de dados, onde eu vou estruturar, transformar o meu dado, criar as features, para garantir que eu consiga criar algum modelo que vai resolver aquela tarefa. E aí eu parto para a tarefa propriamente dita da modelagem. Ou seja, escolher qual é o modelo que eu vou usar, testar vários modelos, treinar esses modelos e obter algumas métricas. Durante o processo de modelagem também é normal que você tenha que voltar para a etapa de preparação, principalmente em engenharia de features. extraísse uma feature, se eu fizesse uma redução de dimensionalidade, será que eu teria um modelo melhor? Então, novamente uma interação aqui entre o processo de modelagem e o processo de preparação. Feita essa modelagem, esse treinamento, a gente parte para uma fase de a avaliação desse modelo, propriamente dito. Então já fiz uma série de interações aqui, eu tenho um modelo que é possível agora de ser validado de fato. E aí eu posso, durante esse processo de validação, novamente recorrer ao entendimento do negócio. Porque não só a métrica técnica, mas será que o modelo está realmente resolvendo o problema ao qual eu discuti lá no início do processo? Então, de novo, uma interação aqui com o entendimento do negócio. Mas a partir do momento que essa avaliação foi feita, que houve uma congruência entre a avaliação do modelo e o que eu entendi como problema a ser resolvido, eu posso partir para o que a gente chama de deployment, colocar o modelo para funcionar em produção. Então, essa é a estrutura do CrispDM. E uma coisa importante no CrispDM que a gente usa bastante, é que o CrispDM, se ele for usado com algum método Lean ou método ágil, ele acaba tendo um poder ainda maior de entrega de valor para o cliente, porque o cliente, geralmente, ele não quer esperar muito tempo para ter um modelo inicial, uma baseline, para ter uma resposta. Então, quando a gente alia essa estrutura cíclica e interativa do CrispDM com algumas práticas ágeis, a gente vai entregar isso que a gente chama do fail fast learn faster, ou seja, vamos testar rápido, se a gente tiver que falhar, a gente falha rápido naquele modelo, mas a gente consegue aprender mais rápido e ter uma interação nova para entregar algo melhor para o nosso cliente na próxima interação. Então, é bastante interessante combinar essas duas práticas do Crisp com alguma prática ágil. Um segundo framework, uma ferramenta aqui no caso, ela não tem uma característica tão grande de processo, de metodologia, como o CrispDM, mas ainda assim é uma ferramenta bastante interessante, que é o MLCanvas. O MLCanvas foi criado em 2016, então ele já foi criado numa outra época, pós-Big Data. com uma ferramenta para ajudar a colocar todo mundo na mesma página quando a gente fala de um projeto de machine learning. Esse é o conceito desse campus. Ou seja, se eu tenho pessoas envolvidas nas mais diversas áreas, querendo entender como aquele projeto tá, porque aquelas pessoas são consumidoras do projeto, elas vão entregar dados para o projeto, ou elas vão avaliar o projeto. Todas as pessoas, através desse Canvas, conseguem entender o projeto de uma maneira única, uma maneira concisa. A ideia do Canvas, esse Machine Learning Canvas, ela foi extraída do que a gente chama de Business Model Canvas, que é um canvas, é um formato de divisão gráfica aqui, de bloquinhos, que aqui no caso do Business Model Canvas, você está modelando um negócio. Então, foi a mesma estrutura, só que aqui você está modelando um projeto de Machine Learning. E aí o Machine Learning hoje tem sido bastante usado por profissionais para que possa ser estruturado ou planejado uma iniciativa de Machine Learning para garantir que todos os elementos estejam considerados e entendidos por todas as partes. De novo, garantir essa questão de ter uma visão única sobre o projeto, todo mundo na mesma página. Passando rapidamente aqui sobre a estrutura, sobre como é o Machine Learning Canvas, ele parte de um ponto central que a gente chama que é a proposição de valor, ou seja, quem é o nosso usuário, por que a gente está, quais são os objetivos dele com esse projeto. E aí, baseado nessa proposta de valor, a gente divide aqui como se fossem duas sub-sessões. Uma sessão que trata muito mais da parte dos dados, então, onde está, como é que eu coleto os dados para esse problema, quais são as fontes de dados, quais são as features, quantos modelos eu vou criar para resolver esse problema, vou criar um modelo para o cliente, vou criar um modelo para a categoria de produto. Então, tudo isso aqui está ligado aos dados. E aqui está mais ligado às previsões, ou seja, Que tipo de tarefa de produção você vai fazer? Você vai fazer uma classificação? É uma recomendação? É uma regressão? Eu vou prever um valor? Que decisões você vai tomar sobre isso? Isso precisa ser real-time? Ou seja, eu vou colocar esse modelo em produção. Ou seja, na hora que alguém faz um pedido. Naquele mesmo momento eu preciso entregar uma predição ou isso vai ser feito em lote, ou seja, eu mando uma série de dados à noite, o meu projeto vai processar isso, o meu modelo vai processar isso e vai devolver uma predição no dia seguinte. E como é que a gente vai... entregar esse modelo, deployar esse modelo em produção. Então, ele separa dessas formas dados e mais a parte da tarefa de predição, as questões relativas à predição. E aqui embaixo, como base, a parte de monitoramento. Como eu monitoro o meu modelo de forma constante ao longo do tempo para garantir que esse projeto continue relevante para o cliente. Então, fica aqui na base do Canvas. E aqui um terceiro modelo, esse aqui já mais novo, criado em 2018, que é o AI Canvas.
