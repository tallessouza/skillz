Primeiro, eu já vi muita gente falar que o código nunca pode ter comentário nenhum. O código tem que ser autodocumentado, o código por si só precisa falar tudo e ele não pode ter nenhum comentário. Isso não é verdade, como também não é verdade que a gente precisa ter comentário em tudo. E para a gente entender comentários, a gente tem que entender que comentários não é a mesma coisa que documentação. Se você está adicionando no seu código comentários dizendo o que o código faz, aí é um erro. Porque o código por si só deveria já dizer isso, ou você deveria ter uma documentação separada, uma documentação especializada para dizer o porquê aquela regra de negócio está ali, ter diagramas de sequência, diagramas de caso de uso para explicar como aquilo funciona, principalmente em regras de negócio mais complexas, geralmente a gente tem esses tipos de diagramas, mas comentário não é documentação. Comentário, ele serve muito mais como, geralmente, um aviso para a próxima pessoa que vai olhar aquele código, de por que que talvez alguma coisa foi feita da forma que foi feita, por alguma limitação, por algum problema da biblioteca, por algum erro, ou para sinalizar algum padrão seguido de uma forma diferente, para que a próxima pessoa saiba como agir naquele determinado momento quando ela vai olhar para esse código. Então, cuidado. Comentário é ok, é normal você ter. Documentação não é legal você ter dentro de comentários. Então, por exemplo, um comentário válido, né? Como eu dei o exemplo, é muito comum a gente às vezes usar uma biblioteca e a gente vai usar determinada funcionalidade da biblioteca e aquela funcionalidade está com um probleminha, por exemplo. E aí a gente tem que... Desculpa. A gente tem que escrever o código ali dentro de uma maneira diferente para contornar esse problema. Só que a próxima pessoa que for dar manutenção nesse código, ela não vai entender por que está escrito daquela forma. Então, ela vai achar que foi uma má prática, foi um código escrito de uma forma errada. E por isso é muito comum eu ver em grandes codebases muitas vezes, um código sendo usado de alguma forma e em cima algum comentário, por exemplo esse código foi escrito assim porque a biblioteca X ainda não suporta Y. E aí quem sabe até um link pro GitHub pra, sei lá pro autor, biblioteca X pra dentro das issues e alguma issue específica que fala sobre aquilo, que tá tratando aquilo, uma pull request, e aí a próxima pessoa que for pegar esse código sabe opa, isso aqui foi escrito dessa forma posso ver se a issue já foi resolvida, se o código já foi resolvido, quem sabe atualizar a biblioteca e corrigir esse código. Então esse tipo de comentário é super legal, se você está vendo que você tem que colocar comentários no seu código para explicar algum código que você escreveu ali para baixo, aí talvez esse código poderia ter sido escrito de uma maneira melhor, de uma maneira diferente. Então, não é tão legal. Aí você está cometendo um erro porque documentação já é algo difícil de manter. Documentação escrita em comentários é praticamente impossível de manter. Então, a grande probabilidade é que se você tenha documentação em comentários, a grande probabilidade é que mais pra frente documentação em comentários a grande probabilidade é que mais pra frente pessoas façam alteração do código mas não alterem a documentação em si, e aí ela vai ficando desatualizada ou daqui a pouco alguém remove ali simplesmente e acabou, foi embora, e ninguém mais entende aquele código, então cuida bastante com isso, comentário não é a mesma coisa que documentação, mas comentário sim é super palpável, super aceit
