Basicamente, quando a gente devolve uma resposta para o front-end, ou seja, após o front-end executar alguma rota aqui da nossa aplicação, por exemplo, uma listagem de usuários, uma criação de um usuário, uma remoção e por aí vai, a gente tem vários tipos de códigos numéricos que a gente pode enviar para o front-end para dizer para o front-end, não só pelo texto que a gente envia, mas para comunicar para o front-end se aquela requisição deu certo, se ela deu erro, se ela deu erro, qual o tipo de erro, se ela deu certo, o que o front-end vai conseguir encontrar de informação dentro do corpo da minha resposta, Se ela deu erro, foi um erro por falta de informações que o front-end deu pra gente ou foi um erro inesperado que a gente não esperava que ele fosse acontecer ali por banco de dados fora do ar ou qualquer coisa assim? Ou seja, esses HTTP status codes, eles têm muita importância semântica na comunicação entre front-end e back-end da nossa aplicação. E aí, novamente, se a gente procurar mdnhttp status code, a gente vai encontrar aqui um guia falando um pouquinho sobre os status codes mais comuns que a gente tem dentro de uma aplicação backend. E a gente vai ver que os status codes, eles sempre têm três números, e eles são sempre de 100 a 200, 200 a 300, 300 a 400, 400 a 500, 500 a 600 no máximo. Todos os status codes que eles estão no intervalo de 100 a 199, eles são status codes informativos. Eles não querem dizer muita coisa, digamos assim, eles servem muito mais para trazer algum tipo de informação ao front-end, mas não querem dizer qualquer tipo de sucesso, erro ou qualquer coisa assim. Todos os status codes de 200 até 299 são status codes de sucesso, ou seja, qualquer número que você retorne de 200 a 299, e 200 é um padrão, todas as requisições por padrão devolvem 200 a menos que a gente retorne alguma outra coisa. 200 quer dizer OK. E é um ok meio vazio, é um ok, mas não quer dizer exatamente que tipo de ok, mas logo a gente vai ver um pouquinho sobre isso. Status codes de 300 a 399 são status codes de redirect, ou seja, quando o front-end faz uma requisição para o back-end para alguma rota que não existe mais e foi redirecionada para outra, ele vai receber um status code 301 ou 302, que são os mais comuns. o 301 quer dizer que aquela rota mudou permanentemente, ou seja, o endereço era esse antes, agora é outro. E aí o 302 ele fala, opa, foi encontrado, foi redirecionado, porém de forma temporária. Então geralmente esse redirecionamento é feito de forma automática, mas os status codes continuam sendo 300, geralmente 301, 302. 400 a 499, a gente tem os client error responses, que são erros originados por causa da requisição que foi feita para o back-end. O que eu quero dizer com isso? Imagine que a gente tem uma rota de criação de usuário, e o front-end vai lá e chama essa rota, porém envia o e-mail do usuário vazio. A gente precisa retornar um erro do back-end, um erro client. Por quê? Porque foi informado algo de forma errada, ou seja, o front-end enviou uma informação de uma maneira errada e por isso originou um erro, ou seja, client error, o client aqui a gente pode entender como o front-end, quem está fazendo a requisição para o nosso back-end ali. E nós temos os server error responses que vão de 500 a 599, o mais famoso é o 500, o internal server error, todo mundo já deve ter visto essa mensagem, ou o 502 também, o bad gateway, quem trabalhou com PHP há muito tempo vai lembrar disso aí, que são os erros inesperados. Não é porque o front-end envia uma informação errada, é porque alguma coisa tá cagada no meu back-end.
