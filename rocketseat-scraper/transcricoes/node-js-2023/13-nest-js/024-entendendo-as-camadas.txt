WEBVTT Nessa aula a gente vai começar a entender um pouquinho como que vai ficar no final das contas aqui a nossa arquitetura da nossa aplicação e tudo que a gente vai construir daqui pra frente com o Nest. Então, antes de mais nada, eu até peguei aqui na esquerda pra gente ter como base a própria Clean Architecture, com certeza você já viu esse gráfico aqui, esse visual. E basicamente, a Clean Architecture, assim como a grande maioria das arquiteturas, é uma arquitetura baseada em camadas, uma Layered Architecture. E aqui, como você pode ver, ela tem basicamente quatro camadas principais, sendo as camadas mais externas, as camadas que geralmente interagem com meios externos, como o próprio usuário ou com banco de dados, a camada de infraestrutura, a camada verde sendo a camada que geralmente vai adaptar as requisições, as conversas que acontecem na camada de infraestrutura para a nossa camada mais interna, que é onde fica a camada de aplicação e domínio, onde tem os casos de uso e as entidades da nossa aplicação. Aqui, como a gente está utilizando conceitos de DDD, meio que essa camada vermelha e a camada de entidades caminham muito próximos, porque é a camada onde fica a parte enterprise, a parte realmente de domínio da nossa aplicação. E aqui na esquerda, eu trouxe uma representação desse mesmo círculo da Clean Architecture, porém trazendo um pouquinho mais conectado com a aplicação que a gente está desenvolvendo para a gente entender o fluxo completo de uma aplicação desenvolvida com essa arquitetura. A primeira coisa que a gente tem que entender é que tudo vai nascer, nesse nosso caso, de uma requisição HTTP. A partir da requisição HTTP que vai bater diretamente dentro da camada mais externa da nossa aplicação, ela vai começar através da interface de usuário. Imagina que isso aqui está dentro, eu só não consigo botar dentro que vai ficar muito confuso. Mas a partir do momento que essa requisição bate nessa camada mais externa, que é a nossa infraestrutura, a camada de infraestrutura vai primeiramente enviar esta requisição para algo que a gente já está construindo e já começou a construir aí no último módulo, que são os nossos Controllers. Deixa eu ver se eu consigo botar ele com uma letrinha menor aqui. Talvez assim. Vamos ver se vai dar para enxergar. Beleza, acho que dá para dar uma giradinha nele aqui. aí então a partir do momento que chega a chamada http a primeira coisa que ela vai fazer é bater nos controllers o controller por sua vez vai enviar esta requisição para a próxima camada que vai ser um use case um caso de uso ou seja o caso de uso vai receber ali dependendo da requisição que o usuário está fazendo o nosso caso de uso vai ser ativado esse caso de uso pode enviar uma ou mais chamadas, fazer uma ou mais interações com entidades da nossa aplicação. E aí eu vou botar aqui Entities, já que eu estou colocando tudo em inglês. Além disso, uma outra coisa importante é entender que esse fluxo aqui não é literalmente uma linha uma reta porque aqui dentro dos nossos casos de uso né vai ser comum também que a gente faça operações no banco de dados então por exemplo se eu tô num caso de uso de autenticação do usuário né eu vou ter que deste caso de uso voltar para uma camada mais externa para buscar dados dos meus repositórios. Então, veja aqui, isso não é necessariamente uma linha reta. Então, eu posso, no meu caso de uso, buscar dados no meu repositório que, por sua vez, vai também para a camada mais externa bater diretamente dentro do nosso banco de dados que, nesse caso aqui, a gente está usando, digamos, o Prisma. Então, o Prisma está na camada mais externa, o ORM aqui da nossa aplicação. Então, veja que, mais ou menos, esse aqui é o fluxo. Depois que aconteceu todo esse fluxo, então, chegou no controller, chegou no use case, bateu nas entidades, o use case foi também nos repositórios, buscou os dados. Depois disso, a gente tem o retorno da resposta para o usuário. Então, aqui do controller, a gente vai ter novamente uma volta aqui dentro melhor, eu acho que não dá nem pra colocar dessa forma aqui mas o controller ele vai chamar um outro carinha aqui que fica na mesma camada, podemos dizer assim que é um presenter, né? o presenter ele nada mais é do que uma camada aqui na nossa aplicação que ele vai adaptar a forma que a resposta é enviada pro usuário ali no fim das contas então o presenter vai voltar, digamos assim para o HTTP, que é a requisição que o usuário fez. O HTTP estaria aqui dentro, ele só hoje não está porque não cabia bem direitinho, mas ele vai voltar para o usuário a resposta do controller. Então, mais ou menos, isso aqui é o fluxo geral que a nossa aplicação vai fazer para uma requisição HTTP tradicional. E aí, como eu não acabei explicando muito bem, o que o Presenter vai fazer aqui no final das contas? Imagina o seguinte, quando o usuário faz uma requisição, imagina que eu estou criando um usuário novo, certo? Eu preciso enviar dados para criar esse usuário novo. Então, eu estou criando um usuário, eu envio para ele o nome, o e-mail, a senha, Esses dados são enviados para o controller. O controller que por sua vez faz aqui um monte de interações com as entidades, com repositórios, com mais pra frente outras partes da aplicação, pode ser criptografia... Bom, ele pode conversar com várias camadas aqui ao mesmo tempo. Mas depois disso, depois de todas as operações que o controller fez, ele vai ter algum tipo de output para devolver para o usuário. No caso da criação de um usuário, não necessariamente eu preciso devolver uma informação. Mas imagina que eu estou fazendo uma requisição de listagem de usuários, eu preciso devolver a lista de usuários aqui no final para esse usuário que está requisitando essa lista. Só que devolver essa lista em que formato? Será que eu simplesmente devolvo a lista no formato que ela está salva aqui dentro das entidades, ou seja, com todos os dados que estão contidos aqui na minha entidade de usuário? Será que eu simplesmente retorno as entidades aqui para o front-end, para o usuário, no final das contas, poder acessar? ou será que eu preciso adaptar essa resposta e muitas das vezes a gente precisa adaptar porque não necessariamente eu vou querer retornar por exemplo todos os campos nessa listagem talvez na listagem eu só precise do id do título de cada usuário eu só preciso talvez eu ou eu preciso de algum relacionamento eu preciso trazer algum dado a mais ou seja o presenter ele tem exatamente esse foco em apresentar os dados em pegar os dados que foram resolvidos aqui pelo controller, que foram buscados aqui pelo controller de dentro dos repositórios, entidades, e converter eles para um formato em que a camada HTTP consiga consumir essas informações. Então, mais ou menos, isso aqui é a estrutura que a gente vai estar seguindo, que é exatamente a estrutura da Clean Architecture, como você bem pode ver. Então, a gente tem a camada mais externa, aqui no caso DB seria o Prisma, UI seria o nosso front-end, Web poderia ser uma API REST, External Interfaces aqui poderiam ser APIs de envios de e-mail e por aí vai, Devices aqui é, no caso, um dispositivo móvel, um iOS, o Android da vida, A gente tem aqui Controllers, como eu bem falei, que são responsáveis por receber essas requisições externas, principalmente HTTP. Presenters, que adaptam a nossa resposta dos Controllers para a UI, para fora. Não só dos Controllers, eles podem adaptar qualquer informação que vai para fora. Gateways, que são basicamente formas de a gente adaptar o restante que não é controller, ou seja, se eu tenho um external interface, por exemplo, eu tenho um envio de um e-mail que ele pode acontecer, que ele acontece por um meio externo. Então, o envio de e-mail precisa ser adaptado assim como a gente faz com repositório para os nossos controllers. Então, por isso que a gente tem os gateways. Os gateways são tudo o que adapta a camada mais externa para uma forma que as camadas mais internas consigam conversar com. Então, o gateway pode ser um repositório, pode ser, como eu falei, para parte de um envio de e-mail, para qualquer coisa que precisa ser adaptada e fazer aquele processo de inversão de dependência que a gente bem viu. Tem os casos de uso aqui, que a gente já colocou na nossa camada de aplicação dentro de domínio, e as entidades que estão na nossa camada de enterprise dentro de domínio. Ou seja, se você for ver para essa imagem, agora no momento que a gente chegou, a gente está chegando em uma representação da Clean Architecture aqui dentro do Nest. E os próximos passos que a gente vai fazer aqui agora é realmente dividir, separar cada uma dessas camadas para que seja bem fácil visualizar onde está ficando cada coisa aqui na nossa aplicação. Vai ser um processo bem interessante e bem legal de fazer. Bora continuar!
