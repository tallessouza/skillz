Só que uma das coisas que eu acho que é bom, a gente já tá num momento que faz sentido, é você entender que lá no JWT existem vários algoritmos. O algoritmo mais comum, que é o HS256, você cria o token, você cria o JWT através de um secret, que é uma string, que geralmente a gente bate a cabeça no teclado e bota qualquer coisa. Mas não é o modelo mais seguro. Por quê? Imagina o seguinte... Deixa eu até abrir aqui o Scalidraw... Imagina o seguinte... Você criou um back-end principal da sua aplicação, que é o back-end onde o usuário faz a autenticação. Esse back-end aqui, quando o usuário faz a autenticação, ele gera um JWT. Imagina que em algum momento você tenha um outro back-end na sua aplicação. Vamos supor que você criou um back-end separado, um microserviço, alguma coisa assim, que é um serviço de notificações, por exemplo. E aí, tem algumas rotas aqui dentro desse serviço de notificações que você precisa garantir que o usuário está logado para ele poder chamar essas rotas. O que vai acontecer? Para você garantir que o usuário está logado no serviço de notificações, você vai precisar validar esse token JWT, que foi gerado por esse serviço de autenticação. Só que o secret responsável por gerar esse token JWT está aqui dentro desse serviço. E aí, será que a gente pega e replica esse secret para dentro desse serviço? Está entendendo que isso é um pouco inseguro, porque... Esse serviço aqui não quer gerar novos tokens, novos JWTs, ele não quer fazer autenticação para o usuário, ele só quer validar que o usuário está realmente autenticado. Ou seja, existe um outro algoritmo que a gente vai usar aqui, que é o RS-256. O RS-256 já é um algoritmo muito melhor nesse caso, por quê? Porque o nosso secret não vai ser simplesmente uma chave, uma string. Ele vai ser uma chave privada e uma chave pública. A chave privada pode ser utilizada para a criação de novos tokens. Ou seja, a chave privada vai ficar armazenada somente no serviço que cria novas autenticações, novos tokens para o usuário. Enquanto a chave pública pode estar em todos os serviços. A chave pública pode estar em todos os serviços que precisam validar que o usuário está alugado. Ou seja, a chave pública, como o próprio nome já diz, ela é pública. Não tem problema ela vazar. Se alguém pegar a chave pública, a pessoa não consegue criar novos tokens. A chave privada, sim. Só que a chave pública, como ela foi gerada a partir da chave privada, ela pode ser utilizada, ela pode ser usada para checar se o JWT do usuário é válido, ou seja, se ele foi gerado a partir desta chave privada aqui. Isso é bem legal, né? Então, a gente vai fazer mais ou menos esse fluxo aqui na nossa aplicação agora, daqui para frente.
