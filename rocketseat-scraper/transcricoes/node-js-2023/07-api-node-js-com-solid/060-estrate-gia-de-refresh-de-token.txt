O Refresh Token nada mais é do que um segundo token, um segundo JWT. Então o que acontece? Quando o usuário faz login, a gente cria um JWT para ele, que é visível, o usuário pode acessar o front-end, pega esse JWT, faz as aquisições, como a gente já viu, usa ele realmente para se autenticar, para fazer as próximas operações, e a aplicação gera junto um segundo JWT, que esse tem uma data de expiração maior, por exemplo, 7 dias, 10 dias, 30 dias, que é um JWT específico para renovação do token original. O que eu quero dizer com isso? Imagina que o usuário foi lá, fez login na nossa aplicação, ele recebe um token, e esse token tem uma data de expiração, por exemplo, de um dia. Perfeito. E o back-end vai ver se existe, então, um segundo token gerado para este usuário que serve para recriar o primeiro token, ou seja, revalidar, ou seja, dar acesso novamente para o usuário, criar um novo JWT para ele dentro da nossa aplicação.
