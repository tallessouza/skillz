WEBVTT Nesse projeto, a gente vai utilizar um micro framework para dentro do Node chamado Fastify. O Fastify, caso você já tenha utilizado ou conhecido o outro micro framework mais famoso do Node, que é o Express, é semelhantíssimo a essa outra tecnologia. E o grande propósito dessas duas tecnologias, tanto o Fastify quanto o Express, é trazer pra gente essa parte mais tradicional, onde a gente precisa utilizar na construção de uma API, de uma aplicação web dentro do Node, ou seja, lidar com rotas, lidar com parâmetros, lidar com plugins externos, lidar com cabeçalhos, respostas em JSON, entender requisições em JSON, coisas que, sim, a gente pode construir na mão, mas que não vale a pena todo o projeto a gente construir do zero, por isso a gente vai utilizar o Fastify. O Fastify, claro, a gente vai conhecer as funcionalidades técnicas dele ao longo do desenvolvimento do projeto, mas algumas coisas que são super legais da gente falar do Fastify em comparação com o Express e o porquê que a gente vai estar optando por utilizar o Fastify aqui é basicamente por três motivos. O primeiro motivo é que o Fastify é muito mais bem mantido que o Express. Ou seja, a gente tem muito mais updates acontecendo dentro do Fastify, um time muito mais ativo lançando novas features, dando manutenção, mantendo a comunidade dentro do Fastify, do que hoje dentro do Express. A própria equipe que mantém o Express já falou que não é um projeto o qual eles dedicam muito tempo mantendo. O segundo ponto de qual o motivo de a gente utilizar Fastify é que ele continua sendo uma das opções mais populares dentro do Node. E a sua API é extremamente semelhante ao Express. Então mesmo aprendendo o Fastify, que é uma opção mais atual e mais atualizada, inclusive mais performática que o Express, Mesmo se um dia você precisar trabalhar com algum projeto em Express, pela similaridade da API, você dificilmente vai ter problemas em trabalhar com as duas tecnologias. O terceiro ponto do Fastify é que, além de ser mais performático, como eu já falei, ele está mais pronto para lidar com novas funcionalidades do JavaScript, como, por exemplo, TypeScript, onde ele já tem uma integração direta com o próprio TypeScript, enquanto no Express a gente precisa instalar uma biblioteca terceira, porque o código do Express não é desenvolvido com TypeScript, E o segundo ponto, ele está muito mais pronto para lidar também com o assincronismo do JavaScript mais moderno, ou seja, trabalhar com promises, trabalhar com o async await, como a gente pode ver aqui no primeiro exemplo já de rota aqui do Fastify, a gente já tem uma função utilizando async. Isso é normal em todas as funções do Fastify, a gente inclusive é obrigado a utilizar o async aqui já para permitir a gente trabalhar com o await dentro de todas as rotas da nossa aplicação. Enquanto com o Express, se a gente trabalhar com async await, a gente vai ter que acabar lidando, trabalhando com outras bibliotecas, instalando outras bibliotecas para facilitar o trabalho com erros e tudo mais, coisas que o Fastify faz por padrão de forma totalmente automatizada. Então, o Fastify é hoje a melhor opção caso você esteja procurando um micro framework, que é um framework menor, que não traz muita opinião, digamos assim, quando comparado com outros frameworks do ecossistema Node, que inclusive, mais pra frente, a gente pode utilizar junto em outros projetos como, por exemplo, Nest, Addonis, ou até outros frameworks como esse, O Fastify, a gente considera ele um micro-framer comparado com esses outros frameworks, porque ele não traz uma opinião muito forte, ou seja, ele não traz opinião em como que você vai estruturar suas pastas, em como que você vai chamar os arquivos da sua aplicação, em qual ferramentas você vai utilizar para integrar, se você vai utilizar um GraphQL ou não, se você vai utilizar um framework de testes X ou Y, ele não traz opinião em cima de nada disso. A única coisa que ele traz, então, é a parte de roteamento dentro de aplicações web, essa parte mais simples, mais core de qualquer aplicação que a gente for desenvolver, com o intuito de fornecer uma API dentro do Node. Quando a gente for utilizar frameworks mais opinados, onde a gente fala, como por exemplo o próprio Nest Framework, aí sim a gente tem uma estrutura de pastas a seguir, a gente tem uma convenção em quais ferramentas, quais bancos de dados a gente vai utilizar, não podendo fugir muito dessas convenções. Mas para quem está aprendendo Node do zero, com certeza o melhor caminho é desenvolver com um micro framework. E também, quem sabe, em aplicações no futuro não vale a pena você utilizar frameworks mais opinados como o próprio Nest, como o próprio Addons... E mais para frente a gente explica quando você vai escolher entre um e outro, mas agora, para o momento, para essa aplicação, a gente vai utilizar Fastify que vai ajudar, vai facilitar um monte aqui no desenvolvimento do nosso app. Então, bora lá!
