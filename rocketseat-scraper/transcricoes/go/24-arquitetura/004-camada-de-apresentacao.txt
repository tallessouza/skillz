A gente já viu nas outras aulas um pouquinho da teoria sobre cada camada, inclusive um pouquinho sobre a camada de apresentação, só que agora vamos focar um pouquinho nela, como a gente entende essa camada e o que ela deveria fazer no nosso projeto. Então, a camada de apresentação, o que seria a camada de apresentação? Então, o que seria a nossa view aqui no nosso projeto? Então, basicamente ela é responsável por exibir as informações ao usuário e receber suas interações. Então ela é que vai se comunicar diretamente com o usuário, podemos assim dizer. Então toda a formatação de dados, se a gente vai formatação para um tipo X, para um tipo Y, por exemplo, para JSON, para algum outro formato, pra XML, se eu vou formatar isso num arquivo, se eu vou formatar isso pra uma API REST, pra um CLI, pra uma CI, pra um GraphQL, pra qualquer outro tipo, é a Vue que vai fazer essa definição. E ela que vai saber definir e se comunicar com cada tipo de cliente ou cada tipo de requisição. Então, todo o tratamento de dados para mostrar ao usuário é feito na View. Então, vamos supor que eu tenho uma informação que eu não quero mostrar, é na View que eu vou definir e colocar um JSON, um tracinho no JSON, isso eu vou mostrar para vocês, por exemplo, ou algo do tipo. Então, é nela que eu consigo definir o que eu mostro ou não, ou o que o usuário me manda ou não, tá? Nela a gente define essa comunicação direta com o usuário. Então qual é o objetivo dela? Basicamente é mostrar os dados ao usuário, geralmente vindo da model via o controller, como a gente já havia falado nas outras aulas. Coletar entradas do usuário, então cliques, digitação, ações, então depende também de onde você está utilizando o seu projeto. Se for numa CLA é coletar a entrada via digitação, que ele digitar os parâmetros que ele digitar, os parâmetros que ele enviar nas flags, ações, dependendo se você estiver utilizando front-end, uma página web, cliques, a mesma coisa, e assim por diante. Ela emite eventos ou sinais que o controller usará para agir, então é ela que vai se comunicar com o controller para poder enviar os eventos ou fazer as comunicações diretas, e ela tem que se manter desacoplada da lógica de negócio, ela não tem que saber nada que o projeto faz. Então, nada que acontece dentro do projeto, como é feito, como a regra é criada, como o objeto é tratado, ele não deve saber de nada. Então, ele não toma decisões nenhuma sobre a informação. Ele simplesmente mostra de alguma forma. Então, não devem conter regras de negócio dentro dela e é muito importante esse ponto. Então, a Vue não deve conter lógica de negócios nem manipulação de dados. Então sua responsabilidade é apenas renderizar a informação recebida e capturar a interação do usuário. Apenas isso. A gente não pode tratar dados que veio do banco de dados, fazer validações ou algo assim diretas, tá? Então toda lógica de cálculo ou transformação de dados deve ser feita delegada ao controller ou ao model, sempre. Então ela não deve ter controle de nada disso, ela só deve renderizar. A Vue lida com eventos como cliques, envio de formulários ou entrada de texto, como a gente já viu, trouxe apenas alguns outros exemplos para você. E essas interações são encaminhadas a um controller, que decide como processá-la e qual forma fornecer. Então ela envia essas informações para o controller, o controller faz o que tem que fazer, faz a camada de dados, atualiza a camada de dados, ou faz o que tem que se comunicar no geral, faz todas as regras volta a informação pra view, a view processa essa informação pra ver como vai mostrar ao usuário e retorna ao usuário é basicamente essa a ideia dessa camada é apenas uma forma de tratar o dado pra ser mostrado de uma certa forma, de um dado específico e
