A arquitetura de software basicamente nos ajuda a definir como as partes de um sistema se organizam e se comunicam. Então basicamente ele vai definir como cada partezinha do nosso código vai se comunicar com alguma outra parte do código. Então como eu consigo criar uma parte do código que se comunica com o controller, com aquele método X, com aquele método Y, com o banco de dados, com uma requisição externa, com tipos diferentes de recebimento de dados, vamos supor um GraphQL, um GRPC, uma CLI, uma CI Jenkins, como a gente consegue criar partes do nosso código diferentes para cada forma diferente de chegar aos dados na nossa aplicação. Então tudo isso é arquitetura e como a gente vai definir o nosso projeto que vão definir. Então ela basicamente orienta a divisão de responsabilidades, ela promove modularidade, ela promove clareza e principalmente a facilidade de manutenção, tá? Então, com uma boa arquitetura, o sistema se torna mais calável, testável e adaptável a mudanças. Então, é muito importante a gente ter uma arquitetura definida, porque o projeto ele tende sempre a crescer. É importante ela ser mantida, porque o projeto continua funcionando perfeitamente, e sempre que alguém for dar manutenção, vai fazer sentido a manutenção que ela vai fazer. Então o que é arquitetura, né? No fim das contas, arquitetura ela é estrutura geral do sistema, então ela que organiza como os componentes são diferentes e como porque eles são diferentes e como eles interagem como eles colaboram, como você consegue fazer um projeto totalmente que faça sentido. Que você consiga conectar as partes e faça sentido essa conexão. Então, ela garante que o sistema seja eficiente, escalável e fácil de manter ao longo do tempo. E ela que define a estratégia que vai impactar o sistema no longo prazo. Então, a nossa arquitetura tem que ser muito boa a longo prazo e não a curto prazo. Geralmente no curto prazo as arquiteturas são até difíceis de criar e difíceis de manter. Porque isso a gente vai falar mais pra frente. Mas basicamente você começa a ter uma arquitetura que você vai criar ela muito modular e muito perfeita. Só que no curto prazo é muito difícil você criar toda a estrutura dela. Só que você começa a ter ganhos no longo prazo porque ela foi bem estruturada, tá? E o que é design de software, né? Então, qual a diferença pro design de software? O design de software, ele se concentra nos detalhes da implementação. Então, na arquitetura, eu defino como tudo vai se complementar e tudo vai se conversar, mas no design de software, eu defino como as coisas vão ser feitas no detalhe. Então, como cada componente é construído, o que cada componente vai resolver e assim por diante. Então, enquanto a arquitetura estabelece a estrutura geral do sistema, o design detalha as decisões que tornam a estrutura prática. Então, a arquitetura, basicamente, eu defino quais módulos eu vou ter dentro do meu projeto, quais camadas eu vou ter dentro do meu projeto e como elas vão se comunicar. Agora, o design de software é como ele vai ser feito essa comunicação. Eu vou utilizar interfaces? Eu vou utilizar herança? Eu vou utilizar o que dentro da minha estrutura do meu projeto? Quais os nomes do padrão de arquivo? Como que a gente vai colocar cada arquivo para se comunicar? Quais os padrões de nome? Quais os padrões de arquitetura ali dentro que eu vou colocar dentro dos meus arquivos? Isso dentro, que eu vou colocar dentro dos meus arquivos. Isso é o design de software que define, tá? Então, quais são as diferenças aqui entre arquitetura e design, né? Então, a arquitetura, ela define a visão geral do sistema, enquanto o design define como cada componente vai ser implementado. Então, como eu vou implementar cada partezinha do meu código, mas a visão geral é a arquitetura que define. Ele foca na comunicação entre os principais componentes, além da escalabilidade e robustez. Então, como eu vou escalar meu projeto? Como eu vou deixar ele mais robusto? Como eu vou separar cada componente? Como cada componente vai ser seguro, vai ser escalável? Como eu vou definir a comunicação entre eles? É a arquitetura que define. Enquanto o design, ele foca em resolver os problemas específicos de cada um de forma eficiente. Então, como eu vou utilizar, se eu vou utilizar interfaces, se eu vou utilizar métodos, se eu vou utilizar implementações, o que eu vou utilizar para comunicar entre eles, é o design que vai definir. Agora, como eles vão se comunicar em uma visão geral é a arquitetura. Um exemplo de arquitetura é você escolher o MVC para organizar seu projeto. Isso é um design e isso é uma decisão arquitetural. Agora, se você define como o método do controller vai interagir com o seu model, isso é uma decisão de design. Então, isso você está definindo o design da sua aplicação e aí muda a diferença, é aí que tem a diferença do design para a arquitetura. É importante a gente conhecer esses desenhos e essas definições.
