WEBVTT Fala pessoal, sejam bem-vindos para mais uma aula de Go aqui na Rocketseat. Agora a gente está iniciando o nosso módulo 4, que nesse módulo a gente vai aprender a como integrar aplicações Go com bancos de dados. Como muita coisa em Go, a gente tem um pacote na estrada de lib para fazer essa integração. Esse pacote é o pacote database-sql, a gente tem a documentação dele aberta aqui e eu quero iniciar esse módulo justamente vendo a documentação. Só que esse pacote é um pouco diferente do pacote net-htcp, porque o pacote net-htcp ele já existe por si só, ele já funciona. Nesse caso aqui você não consegue fazer uma conexão com o banco de dados usando só o pacote database-sql, você precisa de um driver, e o driver nada mais é do que um programa que vai fazer a comunicação de fato com o banco de dados, ele vai implementar o protocolo de comunicação do seu banco de dados escolhido. Então por exemplo, se você for usar SQLite, o seu driver vai implementar o protocolo do SQLite, se você for usar Postgres, o seu driver vai usar o protocolo Postgres, e assim sucessivamente. Só que lá quando a gente estava estudando sobre a HTTP, a gente viu que é muito legal a gente ter os pacotes que são compatíveis com a biblioteca padrão de Go. Aqui quando a gente está falando em SQL, isso também é verdade, normalmente as pessoas preferem pacotes que são compatíveis com as estradas de lib de Go, que é a database-sql. Porém em alguns casos, e a gente vai ver esses casos, é interessante fugir um pouco dessa compatibilidade, porque a gente tratando de banco de dados, a gente ganha alguns benefícios ao fugir dessa compatibilidade com a biblioteca padrão de Go. Mas a gente vai ver isso com mais calma, quando a gente chegar lá eu vou explicar isso certinho. Beleza? Vamos começar a ler então aqui a documentação para a gente entender o que é esse pacote. O pacote SQL fornece uma interface genérica para bancos de dados SQL, ou semelhantes a SQL. Aqui a gente vai ver uma diferença de princípio entre o pacote database-sql e o pacote nethttp. O pacote nethttp é uma implementação do protocolo HTTP. Aqui o pacote SQL, ele é uma interface genérica para banco de dados. Então você vai ter que ter um driver que faz a implementação do protocolo do banco de dados específico mesmo, enquanto que o pacote SQL é só a interface. Só que não só isso, o pacote SQL também implementa algumas funcionalidades para a gente. Por exemplo, se eu vier aqui no tipo DB, olha só o que a gente tem aqui na descrição dele. DB é um identificador de banco de dados que representa um conjunto de zero ou mais conexões subjacentes. É seguro para uso simultâneo por várias goroutines. Ou seja, esse pacote aqui, ele já implementa para a gente um pool de conexões. Um pool de conexões é basicamente várias conexões que a gente tem abertas com o nosso banco de dados. Então ele já faz essa habilitação para a gente, o nosso driver não vai se preocupar com isso. Apesar que tem alguns drivers que tem os próprios pools de conexões e a gente vai ver porquê depois. Mas, isso é uma coisa que o pacote já implementa para a gente. O que o pacote não vai implementar de fato é o protocolo específico de cada banco, porque aí fica a critério dos drivers, até porque tem muito banco de dados e tem outro problema, os bancos de dados são em constante evolução, então tem breaking changes e tudo mais e isso seria um problema para ficar implementando isso na biblioteca padrão de Go. Então esse trabalho de implementar os protocolos dos bancos é terceirizado para os drivers. Mas agora vamos ver quais são as interfaces que esse pacote oferece para a gente. Então se eu baixar aqui por interface, a gente vê que a gente tem duas interfaces, a Results e a Scanner. Vamos começar pela Scanner, que ela é muito importante. Basicamente a Scanner é uma interface utilizada por ROSE.scan, que está logo aqui em cima. O que é a ROSE.scan? É um método no tipo ROSE, que basicamente são as linhas, as entradas do nosso banco de dados. E olha só o que a documentação diz, scan copia as colunas da linha atual nos valores apontados por destino. No caso destino aqui é essa variável DEST. O número de valores em destino deve ser igual ao número das colunas em linhas. Ou seja, vamos supor que o seu banco de dados está retornando três colunas, ID, Nome e Data de Nascimento. Aqui na Destination a gente teria que ter três destinos, que seriam o ID, o Nome e a Data de Nascimento. E aí olha só o que o Scanner faz para a gente. O Scanner converte colunas de linhas do banco de dados nos seguintes tipos comuns de Go e tipos especiais fornecidos pelo pacote SQL. Então automaticamente já vai funcionar o tipo String, Slash Bytes, Inteiro, Inteiro 64, Booleano, Float, Interface, enfim, todos esses tipos aqui. E também alguns tipos que o próprio pacote SQL fornece para a gente, como por exemplo o tipo Null Time. Se eu dar uma pistola aqui, olha, Null Time, a gente tem esse tipo, e basicamente ele é um wrapper do tipo Time.Time, é uma forma da gente interagir com timestamps do banco de dados que podem ser nulas. E é aí que entra essa interface Scanner, porque por padrão a gente só consegue converter esses tipos aqui. Mas a gente pode converter qualquer tipo do banco de dados para uma estrutura de dados em Go, usando a interface Scanner. E a gente também tem uma interface para o caminho oposto, ou seja, a gente converteu um tipo em Go para o tipo do nosso banco de dados. E para isso a gente vai para um outro pacote que é o Database SQL Driver. Aqui a gente tem uma interface chamada Valuer, que basicamente transforma um tipo de dado em Go para um valor para o banco de dados. Então ao implementar essa interface Valuer e a interface Scanner, a gente consegue converter basicamente qualquer tipo de dado de um banco de dados para um tipo de dado em Go. E é aí que as interfaces são usadas para fazer essas conversões. Além disso, esse pacote Driver aqui, ele oferece muitas e muitas interfaces, que são as interfaces que os drivers de Go têm que implementar para poderem ser utilizados pelo pacote Database barra SQL. E só para a gente finalizar, vamos ver o que a interface estava faltando do pacote Database SQL, que é a interface Resultado. Basicamente ela é um resumo de um comando SQL que foi executado. E ela só tem dois métodos, o LastInsertedId, que foi o último id que foi inserido no banco de dados, e o RowsAffected, que é a quantidade de rows que foram afetados por causa daquela query que a gente rodou. Por padrão ela permite que a gente retorne um erro aqui, mas na grande maioria dos drivers esses erros não são utilizados. Você tem que ver a documentação do driver que você estiver utilizando, mas normalmente esses erros aqui podem ser ignorados, porque muitas vezes os drivers não retornam erros aqui. E aqui a gente também começa a ver um problema desse pacote SQL, porque os bancos de dados são muito diferentes uns dos outros. Então, por exemplo, em Postgres a gente não tem esse conceito de LastInsertedId, o Postgres não retorna essa informação para a gente. Então os drivers tem que fazer algum workaround, alguma gambiarra, basicamente, para conseguir implementar essa interface.
