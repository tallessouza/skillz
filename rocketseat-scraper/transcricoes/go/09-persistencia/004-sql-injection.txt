Esse é um ataque muito conhecido, muito popular e que infelizmente muitos serviços ainda sofrem com eles. Apesar de atualmente ele já ser extremamente facilmente compatível. Só que para a gente entender como lidar com ele, a gente primeiro precisa entender o que é SQL Injection. SQL Injection é basicamente quando um attacker consegue infiltrar um SQL que não era esperado dentro da sua query. Isso era muito comum em PHP antigamente, quando o PHP era usado só para fazer static HTML. Então, isso era muito comum de acontecer. Vamos pegar de exemplo essa nossa query aqui. O que a gente está fazendo? A gente está fazendo SELECT star, ou seja, seleciona tudo, dá a tabela FULL, aonde o id é igual à interrogação. Ou seja, aqui poderia ser até minúsculo, inclusive. O que é essa interrogação? A gente viu na última aula que ele é um placeholder. Ou seja, o nosso driver do banco de dados vai substituir essa interrogação aqui pela variável que a gente quiser. No caso, vai ser substituído pelo valor 1. Só que um usuário mal intencionado, um hacker, ele pode tentar quebrar esse banco de dados. Para demonstrar isso, vamos criar uma query que a gente deleta um usuário do nosso banco de dados. Então, eu vou vir aqui e vou chamar de DELETE SQL, que vai ser igual a uma string. Para fazer isso, a gente já sabe como é que é, eu vou fazer DB.exec, vou passar minha query, que vai ser essa DELETE SQL, e meus argumentos, vamos supor que o argumento seja 1. Isso aqui vai retornar um resultado e um erro, por enquanto eu não estou interessado no resultado. Só quero saber se vai dar um erro, e se por acaso der um erro, eu vou causar um pânico. Então, agora eu vou rodar esse código aqui, para ver o que vai acontecer. Primeiro, eu vou deletar o nosso banco de dados, eu vou rodar o código, abre-se tudo certo, criou o nosso usuário Pedro, e depois ele foi deletado. Então, como aqui não deu nenhum erro, a gente sabe que esse usuário foi deletado. Agora, imagina que ao invés de usar um placeholder aqui, ou seja, ao invés dessa interrogação, eu fizesse algo mais ou menos assim, fmt.sprintif, ou seja, formatar uma string diretamente, e ali, eu vou passar um placeholder, que no caso vai ser um %s, vou passar uma string qualquer. E aí, vamos supor que a gente tenha um input do nosso usuário, que é uma string, por exemplo, um formulário HTML, que esse usuário está enchendo logo nesse sentido. E esse usuário, ele só deveria ter permissão para deletar o usuário de id 1, então ele vai lá, porque ele é um bom usuário, ele vai colocar que o input dele é igual a 1. Nesse caso aqui, não vai dar nenhum erro, a única diferença é que aqui, ao invés de passar esse 1 como argumento, a gente só vai passar a nossa carry diretamente, porque o nosso argumento, ele já vai estar direto dentro da carry. Então, a gente só precisa colocar esse argumento aqui, que é o input, e ele vai ser substituído diretamente aqui dentro, nessa %s. E aí, se eu rodar esse código, vamos ver o que vai acontecer, vou apagar o banco de dados, vou rodar de novo, olha só, nenhum erro. Mas é aqui que está o grande vilão da história, é por causa disso aqui que acontece SQL Ingestion. Então, gente, nunca usem um placeholder numa string de uma carry que seja um input de um usuário. Nunca, em hipótese alguma. Ah, mas é usuário autenticado, tem que estar dentro de uma plataforma específica... Gente, nunca faça isso. Sempre, em todas as situações, sem exceção, utilize placeholders, beleza? Por quê? Se o usuário é mal intencionado, olha o que ele pode fazer.
