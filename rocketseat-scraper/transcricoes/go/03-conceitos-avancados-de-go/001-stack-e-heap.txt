Nesse módulo nós vamos abordar conceitos mais avançados da linguagem de programação em Go. E eu quero começar com o que normalmente o pessoal mais se complica, que são os ponteiros. O motivo disso é que tem muita gente que nunca trabalhou com uma linguagem que envolve ponteiros. Pelo menos não diretamente, como por exemplo Python, Java e JavaScript. Só que em Go a gente tem acesso direto aos ponteiros e isso pode ser um pouco complicado. Então vamos já começar falando sobre ponteiros. Só que antes de falar sobre ponteiros diretamente, tem dois conceitos fundamentais que são muito importantes da gente entender. Que é o conceito de memória na stack e memória na heap. Não faz parte desse curso entrar muito a fundo em como essas memórias funcionam lá no hardware mesmo, bem low-level, diretamente nas placas. Mas um conceito abstrato por alto é importante a gente ter. Quando a gente fala sobre stack, a gente está falando da call stack de um thread. Call stack em uma tradução literal seria pilha de chamadas. A call stack é uma estrutura de dados de um formato LIFO, Last In First Out. Ou seja, o último elemento a ser colocado nessa stack vai ser o primeiro a ser retirado dela. É como se fosse uma pilha mesmo. Imagina várias caixas uma em cima da outra. Isso que a gente chama de stack. A gente vai primeiro tirar a caixa de cima, depois a próxima caixa e quando a gente quiser colocar uma nova caixa nessa nossa stack, a gente vai colocá-la no topo. Esse é o conceito de LIFO, Last In First Out. E é nessa stack que a gente salva, por exemplo, os argumentos para uma função, as variáveis locais e outras informações que são necessárias para a execução de uma função. Toda vez que você chama uma função, uma nova caixa, digamos assim, é colocada na nossa stack. E dentro daquela caixa tem todos os dados necessários para aquela função rodar. Quando a gente retorna de uma função, a gente tira aquela caixa do topo da nossa stack. Agora vamos entender o que é a heap. A heap a gente pode usar dinamicamente quando a gente precisa de mais capacidade de armazenamento no nosso programa. Quando uma memória é alocada na heap, ela não é automaticamente liberada quando uma função retorna. É o que acontece quando a gente tira aquela caixinha do topo da nossa stack. Lá a gente tem que fazer uma call para o nosso sistema operacional para ele liberar aquela memória para outros programas poderem usar. Da mesma forma, quando a gente quer alocar memória na heap, a gente tem que fazer uma call para o nosso sistema operacional dizendo o quanto de memória a gente quer alocar na heap. Para quem já trabalhou com o C, isso é bem comum quando a gente faz um malloc ou um free lá na linguagem de programação C. O malloc a gente está pedindo para o sistema operacional criar uma região de memória na heap para a gente, enquanto que quando a gente chama free, a gente está devolvendo aquela área para o sistema operacional. E isso é muito complexo e geralmente causa muitos bugs. Como por exemplo, usar aquela região de memória depois que ela já foi devolvida para o sistema operacional, ou então esquecer de devolver aquela área da memória para o sistema operacional e aí causando o que a gente chama de memory leak, ou vazamento de memória. Que basicamente é uma memória que nunca vai ser devolvida para o sistema operacional. Entre muitos outros problemas. Então o Go, ele abstrai tudo isso para a gente. Go faz pouquíssimas garantias de memória. Ele quase não te dá garantias de o que vai ser alocado na stack ou o que vai ser alocado na heap. Tem várias otimizações. Por exemplo, mapas que são uma estrutura de dados complexos, que a gente vai ver aqui nesse módulo, você imaginaria que seriam sempre alocadas na heap. Mas o runtime de Go, em casos específicos, quando o mapa é muito pequeno, ele pode alocar esse mapa na stack para melhorar a performance do mapa. Mas enfim, de novo, isso não está nada na especificação da linguagem, é simplesmente a implementação que decide isso. E por exemplo, a gente também vai ver isso nesse módulo, tem casos de uso de memória que seriam proibidos em Rust, por exemplo, seria um erro de compilação, enquanto que em C seria undefined behavior. Por exemplo, você retornar de uma função o ponteiro para uma memória que está alocada na stack. Mas em Go você pode fazer isso sem problemas. E o runtime vai decidir se ele vai alocar aquela memória na heap ou se ele simplesmente vai copiar aquela memória para outra parte da stack. Então de novo, gente, esse é um conceito muito complexo. Eu estou só passando por alto aqui para vocês entenderem que existem esses dois tipos de memória. Ainda existem também os registradores, mas em Go a gente quase não fala de registrador, então não faz muito sentido eu falar sobre eles aqui nesse curso. E em Go é uma linguagem garbage collected, ou seja, a memória é gerenciada automaticamente para a gente. A gente não precisa alocar memória nem liberar memória. A gente tem um garbage collector que toma conta disso para a gente.
