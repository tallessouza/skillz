E aí a primeira solução foi virtualização. O que é? Virtualização basicamente é a gente ter um computador completo emulado por software. Beleza? Então pense nele como um computador de mentira. Basicamente um computador de mentira rodando como se fosse um programa dentro do computador de verdade. Então ele vai ter seu próprio sistema operacional, ele vai ter memória RAM, disco rígido, vai ter CPU virtual. Então, por exemplo, para você rodar uma aplicação Linux em uma máquina Windows, você criaria uma VM, uma virtualização, e instalaria o sistema operacional Linux inteiro dentro do seu Windows. Então, uma analogia, uma VM é como se fosse você construir uma casa inteira com fundação, com parede, com telhado, dentro do terreno da sua própria casa, na sua casa principal. Qual que é o problema? Elas são pesadas, então imagina você que está rodando dois sistemas operacionais ao mesmo tempo, então vai consumir muito recurso tudo aqui. Já uma outra solução é a emulação. A emulação, imagina a emulação de videogame mesmo. Então imagina que você tem... É um conceito um pouco diferente, mas ele basicamente traduz instruções de uma arquitetura de CPU para outra. Então, por exemplo, você tem uma arquitetura de algum console antigo, um PowerPC, e aí converte, traduz essas instruções para uma arquitetura x86 de um PC, que é os computadores padrões que a gente tem hoje em dia. o que é computacionalmente ainda mais caro e resolve um problema diferente do Docker. Então essas duas soluções que a gente viu aqui são soluções que resolvem problemas diferentes, beleza? E aí a gente chega no nosso querido Docker, nossos queridos containers. Aqui vem a parte interessante, aqui vem uma evolução inteligente, vamos dizer assim. Os containers foram feitos para serem leves e rápidos. O que é um container? Ele também isola uma aplicação, mas de uma forma muito inteligente. Em vez de a gente criar um sistema operacional inteiro do zero, virtualizar o hardware, o container compartilha o kernel do sistema operacional da máquina hospedeira. Beleza? Então, ele empacota apenas a aplicação e suas dependências diretas, né? Então, bibliotecas, binários, essas coisas. Beleza? Então, a diferença chave entre, por exemplo, virtualização e container. Uma VM, ela é, né, uma casa dentro de uma casa. Então, ela virtualiza o hardware. E tem um sistema operacional convidado completo. Um container, ele é um quarto dentro da casa. Então, ele virtualiza o sistema operacional. tá? Ele usa o sistema operacional do hospedeiro. Só analogia, né? Se uma VM é uma casa inteira, um container, ele é como você alugar um quarto mobiliado na sua casa principal. Então, o quarto, o container, ele vem com os seus próprios móveis, a sua aplicação, as suas dependências, mas ele usa a estrutura principal da casa, a fundação, a eletricidade, o encanamento, já que seria o kernel do SO da sua máquina. Beleza? Então, é basicamente isso. E quando que a gente vai utilizar o Docker?
