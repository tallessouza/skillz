вқҜ # Building a skill-creator for Claude: the definitive guide               
                                                                            
  **Claude's Agent Skills system вҖ” instruction folders with a SKILL.md file 
   вҖ” represents a paradigm shift in how AI agents acquire domain            
  expertise.** The description field in SKILL.md frontmatter is the single  
  most important factor determining whether Claude activates a skill, yet   
  Vercel's evaluations found skills went uninvoked **56% of the time** with 
   default descriptions. This guide synthesizes Anthropic's official        
  documentation, Manus AI's context engineering lessons, academic research  
  on tool selection, and hard-won community experience into a comprehensive 
   blueprint for building an optimized skill-creator system. Skills         
  launched October 16, 2025 and became an open standard (agentskills.io) on 
   December 18, 2025 вҖ” they now work across Claude, Manus, GitHub Copilot,  
  and OpenAI Codex.                                                         
                                                                            
  ---                                                                       
                                                                            
  ## How skills actually work under the hood                                
                                                                            
  Every skill is a directory containing a `SKILL.md` file with YAML         
  frontmatter and markdown instructions. Claude discovers and loads skills  
  through a **three-level progressive disclosure** architecture вҖ” the core  
  design pattern that makes the entire system viable:                       
                                                                            
  **Level 1 вҖ” Metadata (~100 tokens per skill).** At startup, Claude loads  
  only the `name` and `description` from every installed skill's YAML       
  frontmatter into its system prompt. This is the "table of contents" вҖ”     
  just enough for Claude to know when each skill should be used without     
  consuming significant context.                                            
                                                                            
  **Level 2 вҖ” Instructions (<5,000 tokens).** When Claude determines a      
  skill is relevant to the user's request, it reads the full SKILL.md body  
  via bash. The frontmatter is stripped; only the markdown instructions     
  enter context.                                                            
                                                                            
  **Level 3 вҖ” Resources (effectively unlimited).** Scripts, reference       
  files, templates, and assets in the skill directory are accessed only     
  when referenced during execution. Critically, when Claude runs a bundled  
  Python script, **the script's source code never enters the context        
  window** вҖ” only its output consumes tokens.                               
                                                                            
  This architecture means you can install dozens of skills with near-zero   
  overhead. The `name` and `description` fields in frontmatter aren't just  
  metadata вҖ” they're the entire selection mechanism. Skills are embedded as 
   XML in Claude's `Skill` tool definition:                                 
                                                                            
  ```xml                                                                    
  <available_skills>                                                        
    <skill>                                                                 
      <name>pdf</name>                                                      
      <description>Extract and analyze text from PDF                        
  documents...</description>                                                
      <location>user</location>                                             
    </skill>                                                                
  </available_skills>                                                       
  ```                                                                       
                                                                            
  There is **no algorithmic routing or intent classification**. Claude uses 
   pure LLM reasoning via its transformer forward pass to match user intent 
   against skill descriptions. This means description quality directly      
  determines triggering accuracy.                                           
                                                                            
  ---                                                                       
                                                                            
  ## The SKILL.md specification and required structure                      
                                                                            
  ### YAML frontmatter fields                                               
                                                                            
  The file must begin with YAML frontmatter between `---` markers. Two      
  fields are required:                                                      
                                                                            
  | Field | Required | Constraints | Purpose |                              
  |-------|----------|------------|---------|                               
  | `name` | Yes | Max 64 chars, lowercase letters/numbers/hyphens only,    
  kebab-case | Becomes the `/slash-command` in Claude Code |                
  | `description` | Yes | Max 1024 chars (Claude Code) / 200 chars (API),   
  no XML tags | **Primary mechanism** Claude uses to decide when to invoke  
  |                                                                         
  | `license` | No | вҖ” | License identifier (e.g., `MIT`, `Apache-2.0`) |   
  | `compatibility` | No | 1-500 chars | Environment requirements |         
  | `metadata` | No | Key-value pairs | Custom: `author`, `version`,        
  `category`, `tags` |                                                      
  | `allowed-tools` | No | Comma-separated, supports wildcards | Restricts  
  tools: `"Read, Grep, Glob"` or `"Bash(git:*)"` |                          
  | `context` | No | `fork` | Runs skill as isolated subagent |             
  | `disable-model-invocation` | No | Boolean | `true` = only user can      
  invoke (for dangerous ops) |                                              
  | `user-invocable` | No | Boolean | `false` = only Claude can invoke      
  (background knowledge) |                                                  
  | `model` | No | Model identifier | e.g., `claude-opus-4-6` |             
  | `mode` | No | Boolean | `true` = behavior modifier (debug-mode,         
  expert-mode) |                                                            
                                                                            
  **Security restrictions**: XML angle brackets (`<` `>`) are forbidden in  
  frontmatter because it appears in the system prompt and could enable      
  injection. Names containing "claude" or "anthropic" are reserved. Safe    
  YAML parsing prevents code execution.                                     
                                                                            
  ### Directory structure                                                   
                                                                            
  ```                                                                       
  skill-name/                                                               
  в”ңв”Җв”Җ SKILL.md              # Required вҖ” keep under 500 lines               
  в”ңв”Җв”Җ scripts/              # Optional вҖ” executable code (output-only       
  context cost)                                                             
  в”Ӯ   в”ңв”Җв”Җ init_skill.py                                                     
  в”Ӯ   в””в”Җв”Җ validate.py                                                       
  в”ңв”Җв”Җ references/           # Optional вҖ” docs loaded into context as needed 
  в”Ӯ   в”ңв”Җв”Җ api-docs.md                                                       
  в”Ӯ   в””в”Җв”Җ detailed-guide.md                                                 
  в””в”Җв”Җ assets/               # Optional вҖ” templates, fonts, icons for output 
      в””в”Җв”Җ report-template.md                                                
  ```                                                                       
                                                                            
  The file must be named exactly `SKILL.md` (case-sensitive). Folder names  
  must use kebab-case. Claude Code **silently ignores** incorrectly named   
  skills with no error message вҖ” this is a documented bug and common        
  gotcha.

  ## Writing descriptions that actually trigger                             
                                                                            
  The description field is where most skills fail. Anthropic's own          
  documentation states: **"Currently Claude has a tendency to               
  'undertrigger' skills вҖ” to not use them when they'd be useful."**         
  Vercel's agent evaluations confirmed this: in 56% of test cases, skills   
  were never invoked even when conditions matched.                          
                                                                            
  ### The "pushy" description pattern                                       
                                                                            
  Anthropic's official skill-creator recommends making descriptions         
  explicitly enumerate trigger conditions. The formula is: `[What it does]  
  + [When to use it] + [Specific trigger phrases] + [Negative boundaries]`. 
                                                                            
  **Weak description (will undertrigger):**                                 
  ```yaml                                                                   
  description: Helps with projects and documents.                           
  ```                                                                       
                                                                            
  **Strong description (reliable activation):**                             
  ```yaml                                                                   
  description: "Analyzes Figma design files and generates developer handoff 
   documentation.                                                           
    Use when user uploads .fig files, asks for 'design specs', 'component   
  documentation',                                                           
    or 'design-to-code handoff'. Make sure to use this skill whenever the   
  user mentions                                                             
    design systems, UI specifications, or developer handoffs, even if they  
  don't explicitly                                                          
    mention Figma."                                                         
  ```                                                                       
                                                                            
  **Production example from Sionic AI (ML experiments):**                   
  ```yaml                                                                   
  description: "GRPO training skill based on external vLLM server using     
  ms-swift.                                                                 
    Usage scenarios: (1) When performing GRPO training with the vLLM server 
   running on a                                                             
    separate GPU, (2) When encountering errors related to                   
  vllm_skip_weight_sync,                                                    
    (3) When encountering OpenAI API response parsing errors."              
  ```                                                                       
                                                                            
  Notice how the Sionic example includes **exact error phrases** вҖ” Claude   
  can then match these against future error messages. This pattern of       
  encoding specific trigger strings dramatically improves activation rates. 
                                                                            
  ### Research-backed description principles                                
                                                                            
  Academic research on LLM tool selection reinforces several practices. The 
   AutoTool paper (2025) found that effective tool descriptions need four   
  elements: **name, functionality summary, input/output schema, and usage   
  constraints**. ToolExpNet (ACL 2025) demonstrated that semantically       
  similar descriptions between different tools cause selection confusion вҖ”  
  each skill's description must clearly differentiate it from similar       
  skills. The "Less-is-More" paper showed that generating "ideal" tool      
  descriptions first, then matching to real tools, improves accuracy вҖ”      
  suggesting descriptions should match the language users naturally employ. 
                                                                            
  ### Description optimization loop                                         
                                                                            
  Anthropic's skill-creator includes a systematic evaluation approach for   
  tuning descriptions:                                                      
                                                                            
  1. Create 20 evaluation queries вҖ” a mix of should-trigger and             
  should-not-trigger scenarios                                              
  2. Save as JSON array with `query` and `should_trigger` fields            
  3. Run each query 3 times against the current description (for reliable   
  trigger rates)                                                            
  4. Split 60/40 into train/test sets                                       
  5. Use extended thinking to propose improved descriptions based on        
  failure analysis                                                          
  6. Iterate up to 5 times, selecting the best description by test score    
                                                                            
  You can also debug triggering directly by asking Claude: "When would you  
  use the [skill name] skill?" Claude will quote the description back,      
  revealing gaps in coverage.

  ## Anatomy of excellent skills: real-world examples                       
                                                                            
  ### The docx skill (Anthropic's production skill)                         
                                                                            
  Anthropic's built-in document skills reveal the gold standard. The `docx`

  ### Creating New Document

  ## Lessons from Manus AI's context engineering                            
                                                                            
  Manus AI (now part of Meta) adopted the Agent Skills open standard in     
  January 2026 while adding proprietary execution capabilities. Their       
  engineering blog, "Context Engineering for AI Agents," reveals several    
  principles directly applicable to skill-creator design.                   
                                                                            
  **Static tool definitions with dynamic masking.** Rather than dynamically 
   adding or removing tools mid-conversation (which breaks KV-cache and     
  confuses the model), Manus keeps all tool definitions static and uses     
  token-level logit masking to constrain which actions are valid at each    
  state. For skills, this means: don't restructure the skill library        
  mid-conversation. The metadata for all skills should remain stable in the 
   system prompt throughout a session.                                      
                                                                            
  **File system as unlimited externalized memory.** Manus treats the        
  filesystem as persistent, externalized context. Skills should reference   
  files that can be loaded on-demand rather than inlining everything into   
  SKILL.md. Compression strategies should be "restorable" вҖ” keep references 
   (like URLs or file paths) even when content is dropped from context.     
                                                                            
  **Todo-list recitation for attention management.** Across ~50 average     
  tool calls per task, Manus creates and updates a `todo.md` file that      
  pushes objectives into the model's recent attention span. Skills handling 
   complex multi-step workflows should similarly include a "recite your     
  current status" step to prevent the "lost-in-the-middle" problem.         
                                                                            
  **Keep failed actions in context.** Error traces are intentionally        
  preserved so the model can learn from mistakes within a session. Skill    
  instructions should explicitly tell Claude to document failures rather    
  than retry blindly.                                                       
                                                                            
  **Anti-few-shot pattern breaking.** Repetitive similar actions in context 
   cause the model to "few-shot itself" into a rut. Skills that involve     
  repeated operations should introduce structured variation in their        
  serialization to prevent pattern lock-in.

  ## Academic foundations for skill design                                  
                                                                            
  Research on LLM tool selection, meta-prompting, and instruction design    
  provides the theoretical backbone for an optimized skill-creator.         
                                                                            
  ### Tool selection research                                               
                                                                            
  The **Voyager** framework (NeurIPS 2023) established the foundational     
  architecture for skill libraries: an ever-growing collection of           
  executable skills with natural language descriptions, retrieved via       
  **embedding similarity**. Each skill's description is converted to an     
  embedding vector; when the agent encounters a new task, it retrieves      
  skills whose description embeddings are most aligned. Voyager validates   
  skills through three feedback channels вҖ” execution errors, environment    
  feedback, and peer review from another LLM вҖ” achieving **3.3x more unique 
   items and 15.3x faster progression** versus baselines.                   
                                                                            
  The **CREATOR** framework (EMNLP 2023) introduced a critical insight:     
  **disentangle abstract skill creation from concrete execution**. Its      
  four-stage pipeline вҖ” Creation вҶ’ Decision вҶ’ Execution вҶ’ Rectification вҖ”   
  separates the "design a tool" step from the "use the tool" step,          
  improving performance on MATH benchmarks. Every skill needs both          
  documentation AND implementation, and a rectification stage allows error  
  correction.                                                               
                                                                            
  **LATM** (ICLR 2024) demonstrated a division of labor: use a powerful     
  model (GPT-4) for skill creation and a lighter model (GPT-3.5) for skill  
  execution, achieving equivalent performance at significantly reduced      
  cost. This maps directly to the skill-creator pattern: invest heavily in  
  creating high-quality skills once, then let standard Claude execute them  
  repeatedly.                                                               
                                                                            
  ### Meta-prompting and optimization                                       
                                                                            
  The **Automatic Prompt Engineer (APE)** approach treats instructions as   
  programs to optimize: generate candidate prompts, evaluate each using a   
  scoring function, and select the best. APE-generated instructions         
  outperformed human-written prompts on 24 NLP tasks. **OPRO** (Google      
  DeepMind) extends this with an optimization loop where the meta-prompt    
  contains previous solutions and their scores, and the model generates     
  improved candidates вҖ” yielding up to **8% improvement on GSM8K and 50% on 
   Big-Bench Hard**.                                                        
                                                                            
  For a skill-creator, these findings prescribe a generate-evaluate-refine  
  pipeline: produce multiple candidate skill descriptions, test them        
  against evaluation queries, and iteratively improve based on the          
  optimization history.                                                     
                                                                            
  **DSPy** (Stanford, ICLR 2024) replaces hand-crafted prompts with         
  declarative signatures specifying input/output behavior, then             
  automatically compiles optimized pipelines. Within minutes, DSPy-compiled 
   programs outperform standard few-shot by **25-65%**. The implication:    
  skills should be defined as declarative specifications (what they do)     
  rather than rigid templates (how to do it), with automatic optimization   
  of both instructions and examples.                                        
                                                                            
  ### Instruction design principles                                         
                                                                            
  The comprehensive "Prompt Report" survey (2024) cataloging 58 prompting   
  techniques establishes that effective instructions should: begin with     
  action verbs, provide explicit context, state constraints clearly, and    
  use structured formats. "Principled Instructions Are All You Need" (2024) 
   found that **positive framing ("do X") significantly outperforms         
  negative framing ("don't do Y")** вҖ” a direct lesson for skill writing.    
  NeurIPS 2024 research on instruction vs. exemplar optimization showed     
  that **combining clear instructions WITH curated examples yields the best 
   results** вҖ” optimizing either alone is insufficient.

  ## Seven critical pitfalls and how to avoid them                          
                                                                            
  **1. The silent naming failure.** Claude Code silently ignores skills     
  with incorrect folder or file names. The name must be lowercase           
  kebab-case (`review-code`, not `Review-Code`). The file must be exactly   
  `SKILL.md` вҖ” not `skill.md`, `SKILL.MD`, or `Skill.md`. There is no error 
   message; the skill simply doesn't appear.                                
                                                                            
  **2. The Prettier line-wrap bug.** When code formatters like Prettier     
  (with `proseWrap: true`) wrap the description field to multiple lines in  
  YAML, Claude Code silently ignores the entire skill. Keep descriptions on 
   a single line or use YAML block scalar syntax carefully.                 
                                                                            
  **3. Monolithic skill bloat.** One community member split a single        
  voice-and-tone skill into three specialized skills (overall voice, public 
   audience, coaches audience) and saw dramatic improvement. Large skills   
  produce generic output because instructions compete for attention within  
  the context window. **Split when a skill exceeds ~300 lines** or serves   
  multiple distinct purposes.                                               
                                                                            
  **4. Speculative skill building.** Anthropic's official guidance: "Don't  
  write skills speculatively. Build them when you have real, repeated       
  tasks." The litmus test: "Have I done this task at least five times? Will 
   I do it at least ten more times?" Skills built without real-world        
  iteration tend to accumulate irrelevant instructions.                     
                                                                            
  **5. ALL-CAPS directives.** Anthropic's skill-creator warns: "If you find 
   yourself writing ALWAYS or NEVER in all caps, that's a yellow flag вҖ”     
  reframe and explain the reasoning so that the model understands why."     
  Explanation outperforms shouting.                                         
                                                                            
  **6. Wrong point-of-view.** Descriptions must use third person ("Extracts 
   text from PDFs") not first person ("I can extract text"). Claude's       
  system prompt uses third person, so matching this style improves          
  activation accuracy.                                                      
                                                                            
  **7. Auto-triggering dangerous operations.** Skills with side effects вҖ”   
  deployments, sending messages, database modifications вҖ” must set          
  `disable-model-invocation: true` so only the user can invoke them via     
  `/skill-name`.

  ## Building the skill-creator meta-skill                                  
                                                                            
  The skill-creator is itself a skill вҖ” a meta-skill that generates other   
  skills. Based on Anthropic's official implementation, academic research,  
  and community patterns, an optimized skill-creator should implement this  
  pipeline:                                                                 
                                                                            
  ### Phase 1: Discovery interview                                          
                                                                            
  The skill-creator asks probing questions to understand the task: What     
  specific workflow does this skill automate? What are the inputs and       
  expected outputs? What are the edge cases? What does failure look like?   
  What trigger phrases would a user naturally use? This maps to CREATOR's   
  "abstract reasoning" phase вҖ” separating design from implementation.       
                                                                            
  ### Phase 2: Description generation and optimization                      
                                                                            
  Generate 3-5 candidate descriptions using the "pushy" pattern. Each       
  should include what the skill does, when to use it, specific trigger      
  phrases, and negative boundaries. Following APE/OPRO methodology,         
  evaluate each against 20 test queries (10 should-trigger, 10              
  should-not-trigger), run each 3 times, and select the highest-scoring     
  description. Iterate up to 5 rounds.                                      
                                                                            
  ### Phase 3: Instruction authoring                                        
                                                                            
  Structure the SKILL.md body following the template above. Use imperative  
  form, explain reasoning behind rules, include 2-3 complete input/output   
  examples, and add error handling. Keep the main file under 500 lines. For 
   complex skills, create a "menu" in SKILL.md that routes to separate      
  reference files.                                                          
                                                                            
  ### Phase 4: Resource bundling                                            
                                                                            
  Identify what should be scripts (deterministic operations better handled  
  by code), references (documentation loaded on demand), and assets         
  (templates, fonts, icons). Scripts should handle operations where code    
  execution is cheaper than token generation вҖ” sorting, validation, file    
  manipulation, API calls.                                                  
                                                                            
  ### Phase 5: Testing and evaluation                                       
                                                                            
  Anthropic's skill-creator uses parallel subagent testing in Claude Code:  
  spawn test cases as subagents, each running the skill against a specific  
  scenario. Evaluate three dimensions: **triggering accuracy** (does it     
  activate for correct queries and stay dormant for incorrect ones?),       
  **functional correctness** (does it produce the right output?), and       
  **performance delta** (does it improve over baseline Claude without the   
  skill?). WorkOS's pipeline adds a "zero delta" check вҖ” if a skill adds no 
   measurable value over baseline, it shouldn't exist.                      
                                                                            
  ### Phase 6: Iteration with contrastive feedback                          
                                                                            
  Following AVATAR's contrastive reasoning approach, analyze patterns in    
  successful versus failed invocations. What description elements           
  correlated with correct triggering? What instruction phrasing led to      
  better outputs? Feed this analysis back into description and instruction  
  refinement.
